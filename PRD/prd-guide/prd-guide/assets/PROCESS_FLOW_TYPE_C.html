PRD 업무 프로세스 — Type C: 시스템/운영 효율화형

기술적 제약과 업무 효율이 핵심인 과제. 고객향 화면이 없거나 최소이며, AS-IS vs TO-BE 비교와 TechSpec의 비중이 높습니다.

SFN Product|플랫폼팀|v1.0 — Type C
핵심 원칙 4가지
01과제의 배경과 문제 정의가 시작이다. 왜 이 과제를 하는가(Why)와 무엇이 문제인가(What)를 먼저 명확히 해야 한다. 해결 방법(How)은 그 다음이다.
02AS-IS vs TO-BE가 먼저. 현재 업무 흐름을 수치로 정의(소요 시간·오류율)한 뒤 개선 후 흐름을 설계한다.
03고객향 화면은 보통 없음. BO가 핵심 화면. 운영자가 데이터를 어떻게 확인·수정하는지가 중심.
04TechSpec 비중이 높음. 데이터 보존 기간·API 호출 빈도·보안 요구사항은 개발팀과 사전 합의 필수.

AS-IS 설명:
{현재 업무 흐름을 자유롭게 서술}
- 담당자: {역할}
- 주요 단계: {1. 2. 3.}
- 소요 시간: {N시간/일}
- 문제점: {수동 작업, 오류 구간 등}

요청:
1. 업무 흐름을 단계/담당자/소요시간/문제점 표로 정리
2. 자동화 또는 효율화 가능한 단계 식별
3. TO-BE 프로세스 제안 (변경 단계 중심)
4. 기술적 요구사항 초안 (보존 기간, API 빈도, 보안)
5. 누락될 수 있는 예외 케이스·엣지 케이스
✅ AI 제안은 운영 담당자와 교차 검토 필수. 실제 업무 흐름과 차이가 있을 수 있음.

규칙:
- TC를 AS-IS(현재 동작 확인) / TO-BE(개선 후 동작 확인) 두 섹션으로 분리
- AS-IS TC: 기존 동작이 TO-BE 배포 후에도 영향받지 않음을 확인하는 회귀 TC
- TO-BE TC: 개선된 동작을 검증하는 TC
- 운영자 권한·접근 제어 관련 TC 별도 섹션으로 분리
- 데이터 보존·삭제 관련 TC 포함
- Open Items 보류 항목은 "보류 TC" 섹션으로 별도 표기

TechSpec 작성
Type C는 TechSpec 내용이 상대적으로 많음. 아래 항목을 반드시 포함:
· 업무 플로우 (AS-IS → TO-BE 단계별 구현 구분)
· API 인터페이스
· 데이터 보존 기간 및 삭제 정책
· 접근 권한 매트릭스 (역할별 허용 액션)
· 감사 로그 (Audit Log) 요구사항
· 성능 기준 (배치 처리 시간, API 응답 시간)
· 장애 시나리오 및 롤백 방안

검토 항목:

4. 장애 시나리오 — 외부 API 실패, DB 타임아웃, 배치 실패 대응
5. 성능 기준 — 처리량·응답 시간·동시 접속 기준 누락 여부
6. 롤백 방안 — 배포 실패 시 데이터 정합성 보장 방법
✅ 보안 요구사항은 반드시 보안 담당자와 교차 검토 후 확정.
