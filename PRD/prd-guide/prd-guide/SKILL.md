# SFN 플랫폼팀 PRD 작성 스킬

차별화상회(smartfoodnet, SFN) 플랫폼팀의 PRD 표준 포맷에 맞춰 문서를 작성·검토하기 위한 스킬이다. 이 포맷의 핵심 철학은 **"PRD가 곧 자동화의 소스"**라는 점이다. 잘 작성된 PRD는 사람이 읽는 문서이자 동시에 TC 자동 생성, JIRA 작업 분해, UI 생성의 직접 입력값이 된다. 따라서 모호함을 본문에 흘려두지 않고 구조화된 테이블에 가두는 것이 가장 중요하다.

## 0. 과제 유형 판별 (가장 먼저 할 것)

PRD를 쓰기 전에 과제가 세 유형 중 무엇인지 판별한다. 유형에 따라 프로세스와 강조 문서가 달라진다.

| 유형 | 이름 | 성격 | 무게중심 문서 |
| --- | --- | --- | --- |
| **Type A** | 신규 서비스/기능 구축형 | 없던 것을 새로 만든다 | 유저스토리 · 정책 · 화면(고객향+BO) · TechSpec 전체 |
| **Type B** | 데이터 기반 기능개선형 | 기존 기능을 데이터로 고도화한다 | 배경(As-Is 한계)·OKR·TechSpec(로직/모델) 중심 |
| **Type C** | 시스템/운영 효율화형 | 내부 운영·시스템을 효율화한다 | BO 화면 · TechSpec · 정책(운영 규칙) 중심 |

판별이 애매하면 사용자에게 한 가지만 물어 확정한 뒤 진행한다. 유형별 상세 프로세스 플로우는 `assets/PROCESS_FLOW_TYPE_{A,B,C}.html` 및 Confluence 메인 가이드 페이지를 참조한다.

## 1. 문서 구조 (1 + 5 트리)

PRD는 **최상위 1개 + 하위 5개** 페이지 트리로 구성한다. 제목 규칙은 `{PJ/OP}YYYYMM_과제명`이다 (PJ=프로젝트성, OP=운영성).

```
{PJ/OP}YYYYMM_과제명           ← 최상위: Why·OKR·Scope·일정·참여자·관련문서
├── 01-1. 유저스토리_과제명      ← PM 소유. TC·JIRA Story의 소스
├── 01-2. 정책_과제명           ← PM+정책. 유저스토리와 병행
├── 02. 고객향지면_과제명        ← PM+디자이너. UI 생성·TC의 소스
├── 03. BO_과제명              ← PM 소유. 운영자 화면 UI·TC의 소스
└── 04. TechSpec_과제명         ← 개발팀 소유(PM 읽기전용). JIRA Sub-task 소스
```

각 문서의 **소유자**와 **자동화 용도**는 고정이다. 문서 맨 위에 📌 라인으로 소유자·용도·작성 시점을 항상 명시한다.

각 문서의 상세 템플릿은 `references/` 아래에 있다. 해당 문서를 작성할 때 그 파일을 열어 양식을 그대로 채운다.

| 문서 | 템플릿 파일 |
| --- | --- |
| 최상위 PRD | `references/00-top-prd.md` |
| 01-1. 유저스토리 | `references/01-1-userstory.md` |
| 01-2. 정책 | `references/01-2-policy.md` |
| 02. 고객향지면 | `references/02-customer-screen.md` |
| 03. BO | `references/03-bo.md` |
| 04. TechSpec | `references/04-techspec.md` |

## 2. 작성 순서 (의존성 기반)

순서를 지키는 이유는 문서 간 참조(AC ↔ 정책 규칙 ↔ 화면 상태)가 끊기지 않게 하기 위해서다.

1. **최상위 문서** — 과제의 Why·목표·범위를 먼저 못 박는다.
2. **유저스토리** — 정책서보다 **먼저** 시작한다. AC(Given/When/Then)가 모든 하위의 뼈대다.
3. **정책** — 유저스토리와 **병행**. 규칙이 확정되면 즉시 해당 AC의 Then 조건을 업데이트한다.
4. **고객향지면 / BO** — 유저스토리·정책을 화면 상태(State)로 구체화한다.
5. **TechSpec** — 위 문서들을 구현 단위로 번역한다. 개발팀이 작성, PM은 읽기 전용.

## 3. 절대 규칙 (포맷 위반 = 자동화 실패)

이 규칙들은 자동화 파이프라인이 깨지지 않도록 하기 위한 것이다. 어기지 말 것.

- **인라인 TBD 금지.** 미확정 사항을 본문에 "(TBD)"로 흘려쓰지 않는다. 각 문서의 **Open Items / 미결 사항 테이블**에 모아 담고, 본문에는 `[TBD]` 상태 표기 + 해당 OI 번호 참조만 남긴다.
- **취소선 금지.** 제외된 범위는 줄을 긋지 말고 Scope 테이블의 `❌ 제외` 행에 명시한다.
- **변경 이력 인라인 태그 금지.** "(11/26 수정)" 같은 인라인 날짜 태그를 쓰지 않는다. 변경 이력은 Confluence 버전 히스토리로 관리한다. (정책 문서의 "정책 확정 이력" 테이블은 예외 — 규칙 확정을 추적하는 별도 장치다.)
- **상태 표기는 lozenge로.** `미확정`/`완료`, `TC 대상`/`TC 보류`, `검토중`/`BE 확인 필요`/`미설계`, `신규`/`수정`/`유지` 등은 상태 lozenge(Confluence status)로 통일한다. (일반 MD에서는 `[대괄호]`로 표기.)
- **참조 ID를 항상 붙인다.** US-01, AC-1, OI-01, S1-A 처럼 ID를 부여해 문서 간 추적이 가능하게 한다.

## 4. 자동화 연계 맵 (왜 이렇게 쓰는가)

각 문서가 어떤 자동화의 입력이 되는지 이해하면 "무엇을 빠짐없이 적어야 하는지" 판단할 수 있다.

| 문서 | 산출 단위 | 연결되는 자동화 |
| --- | --- | --- |
| 유저스토리 AC (Given/When/Then) | AC 1개 | TC 자동 생성 + JIRA Story 분해 |
| 정책 규칙 | 규칙 1개 | AC의 Then 조건 확정 → TC 보류 해제 |
| 화면 상태 목록 (S1-A …) | 상태 1개 | Claude Design UI 생성 + TC 케이스 |
| TechSpec 업무 플로우 / API | Step·Endpoint | JIRA FE/BE/BO Sub-task 생성 |

그래서 **AC와 화면 상태를 빠짐없이 열거**하는 것이 이 포맷의 생명이다. 빠진 상태·AC는 곧 누락된 테스트와 누락된 작업으로 직결된다.

## 5. 검토 체크리스트 (PRD 마무리 전 확인)

- [ ] 과제 유형(A/B/C)이 판별되어 있고 맞는 문서 세트를 채웠는가
- [ ] 최상위 Scope가 포함/제외/고도화 3구분으로 명시되었는가 (취소선 없음)
- [ ] 모든 TBD가 Open Items / 미결 사항 테이블에 모여 있는가 (본문 인라인 TBD 없음)
- [ ] 유저스토리 AC ↔ 정책 규칙 ↔ 화면 상태가 ID로 상호 참조되는가
- [ ] 정책 확정 시 관련 AC의 Then 조건이 업데이트되고 상태가 갱신되었는가
- [ ] 인라인 날짜·변경 태그가 없는가 (버전 히스토리로 관리)
- [ ] 각 문서 상단에 📌 소유자·용도 라인이 있는가

## 6. 사용 예시

**예시 1**
입력: "준코 매장에 자동발주 추천을 새로 붙이는 과제 PRD 만들어줘"
판단: 없던 기능 신규 구축 → **Type A**. 최상위 + 5종 하위 전체 생성. 유저스토리 먼저, 정책 병행.

**예시 2**
입력: "배송요일 변경 유도 로직을 AI 예측 기반으로 고도화하는 기획서"
판단: 기존 기능을 데이터로 개선 → **Type B**. 배경의 As-Is 한계와 OKR, TechSpec의 계산 로직/데이터 모델에 무게.

**예시 3**
입력: "운영자가 쓰는 정산 BO 화면 정리해서 효율화하는 문서"
판단: 내부 운영 효율화 → **Type C**. BO 화면(상태·필드·에러)과 TechSpec 중심, 고객향지면은 생략 가능.
