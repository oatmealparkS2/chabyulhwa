- 출력 언어: 한국어

- 선행 설계: `2026-05-27-skillify-prd-quality-guard-design.md`, `2026-05-27-prd-quality-guard-subagent-verification-design.md`
- 브레인스토밍 출처: `prd-quality-guard-rerun-brainstorm-state` 메모(2026-05-28 일시정지 → 2026-05-29 재개)

## 1. 목적 & 동기

PRD 분석은 일회성이 아니다. 스킬이 갭을 찾으면 PM이 PRD를 고치고, 고친 PRD를 다시 분석한다 — **재실행(회차)**. 현재 스킬은 매 실행이 산출물 2파일(`doc/TC-*.md`·`doc/GAP-*.md`)을 **덮어쓸 뿐**, 회차 간 추적이 없다. 그래서:

- PM이 "내가 고친 갭이 실제로 닫혔나"를 산출물만으로 확인할 수 없다(이전본이 사라짐).
- 같은 갭이 회차마다 GAP-ID가 재배정돼 회차 간 동일성이 깨진다.
- 어느 PRD 버전을 분석했는지 산출물에 흔적이 없다.

동시에 브레인스토밍 중 두 우려가 드러났다 — (1) TC 우선순위를 스킬이 어떻게 판별하나, (2) 불필요한 TC 양산. 이 우려는 **TC가 이중 목적**(① PRD 갭 발견 수단 ② 실제 QA가 실행하는 테스트 명세)을 가지기 때문에 중요하다. 본 spec은 회차 관리와 함께, 이중 목적을 지키는 **TC 양산 방지 규칙**을 못 박는다.

## 4. 회차/버전 관리 설계

### 4.1 회차 판별 — step 2.5 (신규)

step 2(PRD 페치) 직후, step 3(분석) 전에 **회차 판별 단계**를 끼운다(version 비교가 페치 결과를 쓰므로 페치 뒤에 위치):

1. 폴더 하위 `doc/`에 기존 `GAP-<PRD명>.md`·`TC-<PRD명>.md`가 **있는지** 확인.
   - 없음 → **1회차(신규)**. 기존 로직 그대로.
   - 있음 → **N회차(재실행)**. 아래 4.2~4.5 적용.
2. 재실행이면 step 2에서 페치한 PRD의 `version`을 **기존 산출물 헤더의 version과 비교**(결정 5).
   - 동일 version → "PRD 페이지 버전 변경 없음(v_k 그대로). 그래도 재분석할까?"를 사용자에게 확인 후 진행 여부 결정.
   - 상향 → 정상 재실행.

### 4.2 산출물 헤더 — 버전 메타 블록 (신규, 결정 5)

두 산출물(TC·GAP) 최상단에 메타 블록을 둔다:

```
> **분석 회차**: 3회차 (2026-05-29)
> **대상 PRD**: pageId 123456789 · version v5 · lastModified 2026-05-28
> **이전 회차**: 2회차 (2026-05-20, pageId 123456789 v3) → `_archive/2026-05-20/`
```

`version`·`lastModified`는 `mcp__atlassian__getConfluencePage` 응답에서 캡처. 다중 PRD면 pageId별로 한 줄씩.

### 4.3 이전본 아카이브 (결정 2)

기존 `doc/TC-*.md`·`doc/GAP-*.md`를 `doc/_archive/<이전회차날짜>/`로 **이동**한다(복사 아님 — `doc/` 최상위엔 항상 최신 1세트만). 날짜는 이전 산출물 헤더의 회차 날짜를 쓴다.

**시점 주의**: 아카이브는 **delta 탐지(§4.5)가 이전 GAP 파일을 다 읽은 뒤, 새 산출물을 쓰기 직전에** 수행한다. 그래야 delta 탐지가 `doc/`의 이전 파일을 그대로 읽을 수 있다(아카이브를 먼저 하면 delta가 `_archive/`를 뒤져야 함).

```
doc/
├─ TC-<PRD명>.md            ← 최신(N회차)
├─ GAP-<PRD명>.md           ← 최신(N회차)
└─ _archive/
   ├─ 2026-05-20/           ← 2회차 스냅샷
   │  ├─ TC-<PRD명>.md
   │  └─ GAP-<PRD명>.md
   └─ 2026-05-12/           ← 1회차 스냅샷
```

기존 `차별화리뷰Pro-추천인코드/_archive/`(분리 전 통합문서 보관용)와 **이름은 같지만 위치가 다르다** — 회차 아카이브는 `doc/_archive/`. 충돌 없음.

### 4.4 GAP-ID 항구성 + 결번 색인 (결정 3)

GAP-ID는 **회차를 가로질러 불변**이다. 매 회차 재배정 금지.

- **해결된 갭**: 메인 8열 표에서 제거하고, GAP 파일 말미 「결번 색인」 섹션에 stub만 남긴다.
  ```
  ## 결번 색인 (해결/철회된 GAP)
  - GAP-17: [해결 2026-05-29 · pageId 123456789 v5] 리워드 지급 시점 미정의 — v5에서 "결제완료 즉시"로 확정됨
  - GAP-03: [철회 2026-05-20] 오판정 — 실제로는 AC에 명시돼 있었음
  ```
- **잔존 갭**: 같은 GAP-ID로 메인 표에 유지(설명/심각도 갱신 가능).
- **재발 갭**: 한 번 해결됐다가 새 PRD 버전에서 다시 깨진 갭은 **원래 ID를 재사용**하고 결번 색인의 해당 stub을 제거 후 메인 표로 복귀(변경 이력에 "재발" 기록).
- **신규 갭**: 지금까지 쓰인 최대 ID + 1부터. (결번을 메우지 않는다.)

### 4.5 Delta 탐지 + 변경 이력 (결정 1·4)

재실행 시 스킬은 **이전 회차 GAP 파일을 읽어** 새 분석 결과와 매칭한 **delta 초안**을 만들고, 각 항목을 사용자에게 "동일/재발/신규/해결" 4분류로 확정받는다(결정 4 — 자동 탐지 + 사용자 확정).

매칭 휴리스틱(초안용): 위치(요소·화면)·분류·설명 텍스트 유사도. 확정은 사람이 한다 — 자동 매칭은 제안일 뿐.

확정된 delta는 GAP 파일의 「변경 이력」 섹션(결정 2 — 내장)에 회차별로 누적:

```
## 변경 이력
### 3회차 (2026-05-29, pageId 123456789 v3→v5)
- ✅ 해결: GAP-17 (리워드 지급 시점) — 결번 색인으로 이동
- 🔁 재발: GAP-09 (동시 결제) — v5 정책 변경으로 다시 미정의
- 🆕 신규: GAP-25 (추천코드 만료 후 가입 흐름)
- ➖ 잔존: GAP-02, GAP-11, GAP-16 …
### 2회차 (2026-05-20, pageId 123456789 v1→v3)
- …
```

회차 간 diff가 1급 산출물(결정 1)이므로, 변경 이력은 GAP 파일 본문의 **필수 섹션**이다(생략 불가).

## 5. TC 양산 방지 규칙 + 비목표 명시

### 5.1 양산 방지 규칙 (step 3에 규칙으로 추가)

이중 목적상 redundant TC = 실제 QA 인시 낭비다. step 3의 TC 도출에 다음을 강제 규칙으로 박는다:

1. **클래스당 대표 1개** — 동등분할은 클래스당 TC 1개. 같은 동등 클래스 안에서 값만 바꾼 TC 금지.
2. **경계는 별도 경계 TC로** — 경계값은 의도적으로 분리된 TC(클래스 대표와 중복 아님).
3. **조합은 Pairwise로 압축** — 다요인 조합은 페어와이즈로(이미 기법에 존재; 조합 폭발 방지의 1차 수단).
4. **커버리지 매핑 의무** — 모든 TC는 {동등클래스 · 경계 · 결정규칙 · 상태전이 · 유스케이스 경로 · 에러 시나리오} 중 **최소 1개**에 매핑된다. 두 TC가 동일 커버리지 대상에 매핑되고 추가 변별력이 없으면 **병합**한다.

규칙 1·2는 `references/techniques.md`의 동등분할·경계값 가이드에, 규칙 4(커버리지 매핑 의무 + 병합)는 step 3 본문과 `references/schemas.md` 커버리지 매트릭스 설명에 반영.

### 5.2 비목표 ① — 우선순위는 스킬이 다루지 않는다

스킬은 TC 우선순위를 **산정하지 않고, 우선순위 컬럼도 출력하지 않는다.**

**근거**: 우선순위는 영향도(매출·법규·사용자 수)와 발생가능성에서 나오는데, 이 컨텍스트는 PRD에 없고 QA의 릴리스·회귀 컨텍스트에 있다. 스킬이 PRD만 보고 우선순위를 매기면 *권위 있어 보이는 추측*이 된다(가장 위험). 빈 컬럼을 남기는 것도 거부 — (a) 나중에 자동 채움 표류를 부르고 (b) 산출물이 미완처럼 보인다. 우선순위는 QA가 자기 테스트관리 도구에서 자기 컨텍스트로 부여하는 다운스트림 활동으로 둔다.

→ TC 7열 스키마 **변경 없음**(우선순위 열 미추가).

### 5.3 비목표 ② — 갭을 TC로 강제 변환하지 않는다

완전한 TC를 못 쓰는 지점은 TC가 아니라 **갭**이다(핵심 원칙). 갭을 억지로 TC로 만들면 미정의 동작에 가짜 기대결과를 지어내게 되고, QA가 그걸 실행하면 거짓 PASS/FAIL이 나와 목적 2(실제 실행)를 오염시킨다. 갭은 갭 리포트에 "미정의 동작"으로 남기고, PM이 채운 뒤 **다음 회차**에 비로소 TC가 된다.

→ 현재 방식 유지. step 3·5 변경 없음(이 절은 규칙의 명문화 = 회차 관리와의 정합성 확인용).

| 아카이브 이동 실패/덮어쓰기로 이전본 유실 | 낮음 | 높음(이력 1급 산출물 유실) | 이동 전 대상 `_archive/<날짜>/` 비어있음 확인, 같은 날짜 재실행이면 사용자에게 확인 후 진행 |

| GAP-ID 재사용 규칙 혼동(재발 vs 신규) | 중간 | 중간 | §4.4 명문화 + C7이 단조성·dangling 검증 |

## 9. 검증

1. **1회차 회귀**: 신규 폴더에서 스킬 실행 → 회차 로직 미발동, 기존 산출물과 동일 형태(헤더 메타만 추가) 확인.
2. **2회차 시뮬레이션**: 1회차 산출물이 있는 폴더에서 PRD version 상향 가정 → 이전본이 `doc/_archive/<날짜>/`로 이동 · 헤더에 회차/version · 변경 이력 섹션 생성 · 해결 GAP이 결번 색인으로 이동 · 신규 GAP이 max+1 ID 확인.
3. **version 미변경 분기**: 같은 version으로 재실행 → "변경 없음, 그래도 진행?" 확인 프롬프트 발동 확인.
4. **C7 검증**: 결함 주입(헤더 메타 누락 / 결번 색인에 dangling GAP-ID / 신규 GAP이 max+1 아님) → 검증 subAgent가 C7 FAIL로 정확 지목.
5. **양산 방지**: 동등 클래스 값만 다른 중복 TC를 의도 주입 → step 3 규칙 4(병합)이 잡는지 수동 확인.
