# 1. 개념정의
## 1-A. 상품추천

### 영역 A: "구매할 때 됐어요" (재구매 상품)

고객이 이미 구매 이력이 있는 상품 중 구매 주기가 도래하는 상품을 노출한다.

- **대상:** 기존 구매 상품 (신선식품 30일 이내, 가공식품 60일 이내 구매 이력)
- **정렬 기준:** 평균 구매 주기 대비 미주문 기간 차이가 적은 순 (구매 임박 순)
- **구매 임박 표시:** D-day 형태 (D-0 = 오늘 구매 예상, D-3 = 3일 후 예상)
- **구매 주기 산출:**
  - 2회 이상 구매 이력 상품: 고객×SKU별 개인 구매 주기 (구매 간격 중앙값/가중평균)
  - 1회 구매 상품: 해당 카테고리의 평균 구매 주기 적용
- **쿠폰 적용:** 동적 프라이싱 엔진에서 산출된 상품별 차등 할인 쿠폰 부착
- **Phase 1 대응 :** SFN이 룰 기반 구매 주기 기준을 제공하면 Weraser가 추천 결과에 플래그를 부착 (is_repurchase, purchase_urgency_score). 개인별 SKU별 구매 주기 AI 모델링은 Phase 2.
  
> **구현제외** 
- **이탈 방지 로직:** 반복 주문했으나 더 이상 찾지 않는 상품(평균 구매 주기 대비 미주문 기간이 현저히 큰 상품)에 대해 할인 기재 적용 + 우선순위 상향 
- 구매 임박 로직의 기준 데이터는 SFN 내부에서 관리하며, Weraser는 전달받은 기준에 따라 플래그를 찍어 구분한다. 기준 데이터(상품 카테고리별 며칠 주기 추천 등)만 있으면 추가 모델/데이터 없이 플래그 부착 가능. 복잡한 개인별 구매 주기 모델링은 Phase 2.


> **합의사항 (2026.03.16):** 구매 임박 로직의 기준 데이터는 SFN 내부에서 관리하며, Weraser는 전달받은 기준에 따라 플래그를 찍어 구분한다. 기준 데이터(상품 카테고리별 며칠 주기 추천 등)만 있으면 추가 모델/데이터 없이 플래그 부착 가능. 복잡한 개인별 구매 주기 모델링은 Phase 2.

### 영역 B: AI 맞춤추천 상품 

고객이 아직 구매하지 않는 상품 중 메뉴 기반 분석으로 필요할 것으로 판단되는 상품

- **대상 :** 미구매 상품 + 아웃 오브 데이트 상품 (신선 30일, 가공 60일 초과 미구매)
- **정렬 기준:** 5축 통합 스코어링 (§3.3)
- **배지:**
  - `최저가 도전` — 경쟁사 대비 SFN 가격이 저렴한 상품 (판매가 기준 + 고객별 쿠폰 적용 후 기준 이중 비교)
  - `전용 할인` — AI 상품 쿠폰이 발급된 상품에만 표시
- **쿠폰 적용:** 갭 분석 기반 상품 쿠폰 (§4.2)

### 탭 구조 및 데이터 분리

- AI 추천 상품과 재구매 상품은 **별도 탭/필터**로 분리
- Weraser → SFN 추천 데이터 전달 시 각 상품에 `recommendation_type` 필드로 구분 (`new_recommendation` / `repurchase`)
- SFN 내부 재구매 상품 리스트(단순 주문이력 역순 기반)는 SFN 자체 API로 처리하며, Weraser 추천 결과와 별도 운영

## 1-B. 동적 프라이싱 엔진 *(v5 신규 — 2026.03.16 회의)*

### 1-B.1 목적 및 배경
	
	SFN은 상품당 단일 가격만 설정 가능한 시스템 구조이다. 고객별 차등 가격을 구현하기 위해 전 상품 가격을 일괄 인상한 후, 고객별 상품 쿠폰으로 실질 구매 가격을 개인화한다.

- **가격 운영 공간 확보:** 전 상품 30% 일괄 인상 (Up-price)
- **실질 가격 개인화:** 고객별 가격 민감도에 따른 차등 상품 쿠폰 발행
- **GP 목표:** 현재 12% → 13% (초기 +1.33%p, 성공 시 지속 상향)

### 1-B.2 가격 구조 정의

| 용어 | 정의 |
|------|------|
| Base Price | 기존 판매가 (인상 전) |
| Up-price | 인상 후 시스템 등록 가격 = Base Price × 1.30 |
| 실질 구매가 | Up-price − 상품 쿠폰 할인액 (고객이 실제 결제하는 금액) |

> **가격 운영 주체 (v5 합의):** Up-price 이후 상품의 모든 가격 운영(변경·조정)은 **Weraser에서 담당**한다. SFN 시스템에서 단일 가격만 설정 가능하므로, 동적 가격 조정을 Weraser가 일원화하여 관리한다.

### 1-B.3 프라이싱 대상 상품

- **대상:** 고객별 6개월 이내 2회 이상 구매한 전체 상품
- **제외:** 1회 구매 상품 (반복 구매 패턴 미형성, 가격 민감도 높음, 구매 주기 특정 불가)

### 1-B.4 가격 민감도 티어링

고객별로 자신이 구매하는 상품을 **매출 비중(가격 × 수량 × 횟수, 최근 6개월)** 기준으로 25%씩 4개 티어로 구분한다. 전체가 대상이며 상품 수에 무관하게 매출 비중 백분위로 배정.

| 티어 | 매출 비중 | 인상률 | 근거 |
|------|-----------|--------|------|
| **Top** (상위 25%) | 가장 높음 | **인상하지 않음** (단, 동업계 대비 이미 고가 구매 시 +0.3%) | 고객이 가장 민감하게 인지하는 상품. 가격 변동 시 이탈 위험 |
| **Key** (25~50%) | 높음 | **인상하지 않음** (단, 동업계 대비 이미 고가 구매 시 +0.3%) | Top 다음으로 민감한 상품 |
| **Standard** (50~75%) | 보통 | **기존 대비 +1.5%** | 일반 인상 대상 |
| **Tail** (하위 25%) | 가장 낮음 | **기존 대비 +3.5%** | 가격 민감도 가장 낮은 상품 |

> **가설 근거:** SFN 고객 인터뷰 및 경험치에 따르면, B2B 식자재 고객은 자신의 지출 비중 상위 ~30% 상품에만 가격에 예민하고, 하위 상품에 대해서는 상대적으로 둔감하다. 상위 50%(Top+Key)는 가격을 건드리지 않아 이탈을 방지하고, 하위 50%(Standard+Tail)에서 마진을 확보한다.
> 
> **초기 설정값:** 위 인상률은 최초 설정값이며, 운영 데이터에 따라 SFN이 조정. 전 티어 적용 시 예상 GP 인상분은 +1.33%p (12% → ~12.33%). 목표 13% 달성 시까지 단계적 상향 예정.
> 
> **데이터 근거 부재 인지 (2026.03.16 회의):** 위 수치는 경험치 기반 가설이며 정밀한 가격 민감도 파악 데이터가 없는 상태. 설정 후 학습해가며 조정하되, 매출 빠지면 즉시 재조정한다는 합의.

### 1-B.5 예외 처리

| 예외 | 조건 | 처리 |
|------|------|------|
| **SKU 스위칭** | 동일 leaf 카테고리 내 구매 SKU 변동 이력 존재 (A브랜드 → B브랜드 왕래) | **Top 티어로 고정** → 가격 인상하지 않음 |
| **소량 구매 고객** | 구매 상품 5개 이하 | **전체 제외** (극히 일부만 구매 = 가격 이유일 가능성 → 이탈 위험) |
| **경쟁사 가격 상한** | 티어별 경쟁사 가격 대비 인상률 상한 | Top: 5%, Key: 5%, Standard: 10%, Tail: 10% (하드 룰) |

### 1-B.6 구매 후 액션

| 상황 | 액션 |
|------|------|
| 가격 인상 후 **고객이 구매함** | 평균 구매 주기 × 3 (≈ 4회 구매)까지 현 가격 유지. B2B 고객(판매가=공급원가 인식)의 접속 시마다 가격 상승 불만 방지 |
| 가격 인상 후 **미구매 지속** | 기존 구매 가격으로 원복 후, 해당 상품은 추천 로직으로 전환 |

### 1-B.7 신규 고객 기본 쿠폰 정책

전 상품 가격이 Up-price로 인상되어 있으므로, 신규 고객에게는 기존 판매가 수준으로 구매할 수 있도록 기본 쿠폰을 발행한다.

- **쿠폰 산출:** Up-price − Base Price (= 인상분 전액 할인)
- **발행 방식:**
  - 상품 쿠폰 스킴(상품별 할인 금액 정의)을 **전일 배치로 사전 생성**
  - 신규 고객 가입 시 **즉시 발급** (쿠폰 매칭만 실시간 처리)
- **적용 대상:** 전체 상품 (신규 고객은 구매 이력 없음 → 가격 민감도 판단 불가 → 일괄 기본 쿠폰)
- **동일 신규 고객 간 차이 없음:** 모든 신규 고객에게 동일한 상품별 기본 쿠폰
- **SFN 쿠폰 시스템 구조:** 상품 1개에 여러 종류의 쿠폰(천원, 이천원 등) 생성 가능, 다수 고객에게 복수 매핑 가능. 신규 고객용 기본 쿠폰은 별도 스킴으로 사전 생성, 가입 시 자동 매핑.

### 1-B.8 동적 프라이싱과 추천 로직의 충돌 해소

| 충돌 유형 | 상황 | 해소 방안 |
|-----------|------|-----------|
| 미구매 상품 추천 + 동적 프라이싱 | 미구매 상품에 추천 쿠폰 적용 중인데, 동적 프라이싱도 적용? | 미구매 상품은 추천 쿠폰만 적용 (동적 프라이싱 미대상 — 2회 이상 구매 상품만 대상) |
| 재구매 상품 + 가격 인상 | 기존 구매 상품의 가격을 올리면서 동시에 할인 쿠폰 | 실질 가격 = Up-price − 차등 쿠폰. 시스템: 쿠폰 할인, 운영: 가격 인상 |
| 이탈 위험 고객 + 가격 인상 | 이탈 위험 고객의 상품 가격 상승 → 이탈 가속 | 이탈 위험 고객은 동적 프라이싱 제외 또는 인상률 최소화 (Phase 1 룰 베이스) |

> **실효성 우려 (2026.03.16 회의):** Weraser 측에서 동적 프라이싱의 실효성(실제 매출/마진 향상 이어지는지)에 대한 우려 표명. Phase 1에서 A/B 테스트를 통해 검증하고, 미달 시 전략 재조정.

## 1-C. 데이터 교환 프로토콜 *(v5 확정 — 2026.03.16 회의)*

### 1-C.1 SFN → Weraser (인풋 데이터)

| 데이터 | 방식 | 주기 | 비고 |
|--------|------|------|------|
| 고객 정보 (신규/변경) | **SQS 메시지 직접 전송** | 이벤트 기반 (실시간) | 신규 가입, 주소 변경 등. 신규 고객은 수가 많지 않으므로 SQS로 바로 전송 |
| 상품 마스터 (전체 판매 상품) | **S3 적재 + SQS 이벤트 알림** | 일 1회 배치 | 상품 속성, 카테고리, 보관방법 등 |
| 상품별 원가 정보 | S3 적재 | 일 1회 배치 | GP 마진 산출용 |
| 주문 데이터 | S3 적재 | 일 1회 배치 | 주문+주문상품+금액 |
| 장바구니 정보 | S3 적재 + SQS 이벤트 | 일 1회 배치 | 구매 패턴 분석용 |
| 고객 접속 정보 | S3 적재 | 일 1회 배치 | 접속 패턴 참고 |
| FC 고객 영업표지명 매핑 | 수기 엑셀 덤프 | 비정기 (변경 시) | SFN 내부 수기 관리 |

> **SFN 현황 (2026.03.16):** S3 업로드 → SQS 이벤트 발행 구조 확인 완료. SFN 서버가 모노리식이라 Kafka 등 비동기 처리 대신 S3 파일 교환 방식 채택. 상기 데이터 전달은 해당 주 내 완료 예정.

### 1-C.2 Weraser → SFN (아웃풋 데이터)

| 데이터 | 방식 | 주기 | 비고 |
|--------|------|------|------|
| 고객별 추천 상품 리스트 (배치) | **S3 파일 업로드 + SQS 이벤트 알림** | 일 1회 (전일 처리 → 익일 아침) | ~20,000명 × 10개 = ~200,000건. LMS 발송 08:00 이전 완료 필요 |
| 고객별 추천 상품 (신규 고객 즉시) | **API 호출 (SFN API)** | 이벤트 기반 (실시간) | 신규 가입 후 2~3분 내 반환. 기존 캐싱 데이터 우선 (LLM 호출 최소화) |
| 고객별 상품 쿠폰 정보 (배치) | S3 파일 + SQS | 일 1회 배치 | 추천 상품별 쿠폰 금액 포함 |
| 고객별 상품 쿠폰 (신규 고객 즉시) | API 호출 | 이벤트 기반 | 신규 고객 기본 쿠폰 즉시 매핑 |

> **아웃풋 스펙:** 양사 개발자 간 이번 주 중 확정 예정. 최소 포함 필드: `customer_id`, `sku_id`, `recommendation_type` (new/repurchase), `priority_rank`, `score`, `coupon_amount`, `is_lowest_price`, `purchase_urgency_flag`

### 1-C.3 CRM/LMS 연동

- Weraser가 일 배치로 추천 결과 + 쿠폰 정보를 S3에 적재 → SFN이 익일 오전 LMS 배치 발송 (08:00)
- LMS 메시지 구성: 재구매 임박 상품 + 개인화 추천 상품 + 상품 할인 쿠폰 + 장바구니 쿠폰
- Weraser S3 업로드 완료 시 SQS 이벤트 발행 → SFN에서 수신 후 내부 처리

## 1. 분석 전제: 기존 구매 데이터만으로 무엇을 알 수 있는가

SFN이 현재 보유한 데이터는 크게 세 종류다.

- **구매 이력:** 고객 ID × SKU × 수량 × 금액 × 일시 (장기간 축적)
- **장바구니 쿠폰 사용 이력:** 발급 금액, 사용 여부, 사용 시 결제 금액
- **고객 기본 정보:** 배송 주소, 가입일, 배송 타입, 사업자 여부, 마케팅 동의 여부 등. **업종(대분류 11개, 82.9% 커버)·메뉴(카테고리 태그 수준, 66.8% 커버)도 보유하나 식자재 추출에 필요한 상세 수준(개별 메뉴명·가격·식자재)에 미달** → 크롤링 데이터(네이버 플레이스/다이닝코드)가 프로파일링 주요 소스, SFN 자체 업종·메뉴는 보조 용도(cross-validation, 크롤링 실패 시 폴백, 임시 클러스터 배정) [회원정보 데이터 반영]

SFN은 개별 식당의 전체 매출 데이터를 보유하고 있지 않다. 클릭, 조회, 체류 시간 등 행동 로그도 없다. 따라서 Phase 1에서는 **구매 트랜잭션 데이터만으로** 고객의 구매 패턴을 역산하고, 결핍 SKU를 도출하며, 쿠폰 반응도를 추정하는 방법론을 적용한다.

핵심 원리는 단순하다: **"고객의 메뉴에서 필요한 식자재를 역산하고, 그 중 아직 구매하지 않는 SKU를 추천한다."** 고객이 운영하는 '메뉴-식자재'가 추천의 1차 준거 기준이며, 유사 고객군의 구매 패턴은 추천 SKU Pool을 보완하고 용량·주기 추정에 활용하는 보조 지표이다.

## 2. 데이터 분석 방법론

### 2.1 고객 프로파일링 (외부 데이터 크롤링)

구매 데이터만으로는 고객이 어떤 음식점인지, 무슨 메뉴를 파는지 충분히 알 수 없다. SFN 회원정보에 업종(대분류 11개, 82.9%)과 메뉴(카테고리 태그 수준, 66.8%)가 존재하나, 식자재 추출에 필요한 상세 수준(개별 메뉴명·가격·식자재 정보)에 미달하므로 **크롤링 데이터가 프로파일링의 주요 소스**이다. SFN 자체 업종·메뉴 데이터는 보조 용도(cross-validation, 크롤링 실패 시 폴백, 임시 클러스터 배정)로 활용한다. [회원정보 데이터 반영] 배송 주소를 조인키로 외부 소스에서 프로파일을 구축한다.

**처리 흐름:**

```
고객 배송 주소
    → 네이버 플레이스 우선 크롤링 (primary source)
                수집 항목: 메뉴(메뉴명, 가격, 대표/인기 태그),
                   메뉴 Description 내 "식자재 정보" 및 "세트 메뉴 구성 정보",
                   업종

    → FC 판별: SFN 직접 제공 FC 고객 리스트(1차) + 네이버 상호명 기준 공공데이터 비교(2차)
        ※ 1차와 2차는 **합집합** 개념으로 진행 [SFN Comment 반영]

  │   네이버 플레이스에서 대부분 검색 불가 또는 메뉴 미등록.
``` 

**FC 정보 관리 체계:**

SFN이 As-is 기준 FC 고객 리스트를 직접 제공하며(정보공개서 영업표지와 브랜드명 매칭 포함), FC 신규 고객 계약 시 수기로 리스트를 업데이트한다(비정기 이벤트). **FC 고객 영업표지명 매핑은 업데이트 시 수기 엑셀 덤프로 전달하며, 그 외 데이터는 API 또는 배치로 데이터 교환이 필요하다.** [SFN Comment 반영]

> **SFN FC 리스트의 한계 인지:** SFN이 제공하는 자사 프랜차이즈 고객 리스트는 해당 식당이 프랜차이즈라는 점에서 매우 정확하나, **전체 프랜차이즈 브랜드/매장 중 극히 일부에 해당한다.** 따라서 SFN FC 리스트에 없다고 프랜차이즈가 아닌 것은 아니며, 보정에는 한계가 있다. [SFN Comment 반영]

> **자사 FC 브랜드의 메뉴 크롤링 생략:** SFN 자사 FC 브랜드는 이미 SKU 단위로 수요 식자재가 정의되어 있으므로, **별도의 메뉴 정보 취합(크롤링)이 불필요하다.** SFN이 브랜드별로 선정하여 전달하는 SKU Pool을 그대로 사용한다. [SFN Comment 반영]

#### 2.1.1 LLM 기반 메뉴-식자재 추출

크롤링된 메뉴 리스트에서 해당 메뉴를 만드는 데 필요한 식자재를 LLM으로 추출한다. 추출된 식자재는 SFN SKU 카탈로그에 매핑되어 추천 SKU Pool의 1차 기준이 된다.

```
크롤링된 메뉴 리스트
    → LLM 식자재 추출 (1회 처리 후 캐싱)
    → 추출 소스:
       ① 메뉴명에서 직접 노출된 식자재 (e.g., "한우 등심 스테이크" → 한우 등심)
       ② 메뉴 Description에 기재된 식자재 정보 (e.g., "국내산 돼지고기 사용")
       ③ LLM 추론 식자재 (e.g., "김치찌개" → 배추김치, 돼지고기, 두부, 대파, 고춧가루)
    → SFN SKU 카탈로그 매핑 (**leaf 카테고리 기준** 최하위 분류) [SFN Comment 반영]
       ※ 예: 김치찌개의 돼지고기 → L3 기준이면 "가공식품_두류가공품_두부"이지만
          leaf에는 모두부, 순두부, 판두부, 건두부 등 다양 → leaf 수준 매칭 필요
       ※ 기존 고객 구매 데이터를 통해 [메뉴-식자재]별로 매칭되는 leaf 카테고리 정리 가능 여부 확인 필요
    → FC 특성 반영: FC 식당의 경우 반제품 위주 추출 반영
       (e.g., FC 도시락집의 "탕수육" → 식자재 = 냉동 탕수육)

검증 기준 (Phase 1, 1주차):
  ├── SKU 매핑률 목표: 70% 이상

  └── 매핑 실패 시 프롬프트 튜닝으로 보완
```

#### 2.1.2 경쟁사 가격 크롤링 [SFN 피드백 반영: 매출 상위→전체 SKU 크롤링]

상품 쿠폰 할인율 산정(§4.2) 및 추천 우선순위 스코어링(§3.3)에는 경쟁사 가격 데이터가 필수다. SFN 가격이 경쟁사 대비 얼마나 비싼지를 알아야 적정 할인율을 산정하고, 가격 경쟁력이 높은 상품을 우선 추천할 수 있기 때문이다.

**Phase 1 — 전체 SKU 크롤링:**

```
대상: 식봄(Sikbom) + 배민상회(Baemin Sanghoe) 2개 플랫폼
수집 항목: SKU별 판매가, 프로모션/할인 여부, 수집 시점
주기: Daily (상품 추천이 Daily이므로 가격 크롤링도 Daily 필수)
범위: 차별화상회에 등록된 전체 활성 SKU (약 5,011개)
      ※ 매출 상위 SKU가 아닌 전체 SKU 필수:
         못 팔고 있는 상품을 추천해서 팔려는 것이 목적이므로
         전체 상품의 가격 경쟁력을 파악해야 함
```

SKU 매핑이 핵심이다. 경쟁사 SKU와 SFN SKU는 명칭/용량/브랜드가 다를 수 있으므로, 동일 카테고리 + 유사 용량 + 단위용량당 단가 기준으로 매핑한다. Phase 1에서는 룰 베이스 매핑 후 SFN 검수를 거치고, Phase 2에서 자동화한다.

**Phase 2 — 고도화:**

```

가격 변동 추이 분석 → 경쟁사 프로모션 탐지
```

> **폴백 정책:** 경쟁사 가격 데이터가 없는 SKU에 대해서는 **가격할인(상품 쿠폰)을 발행하지 않고 추천만 진행한다.** 경쟁사에 가격 정보가 없는 SKU라면 고객도 가격 비교가 불가능하다는 의미이므로, 가격 할인 없이도 추천만으로 구매 전환이 가능하다. [SFN Comment 반영] 다만 GP 마진 기반 전략적 할인이 필요한 경우(예: 미끼 상품)에는 예외적으로 할인 가능하다.

### 2.2 고객 세그먼테이션 [SFN 피드백 반영: 4→3 세그먼트 통합]

구매 이력 + 고객 프로파일을 결합하여 **3개 세그먼트**로 분류한다. 세그먼트별로 분석 방법과 추천 로직이 완전히 다르기 때문에 이 분류가 선행되어야 한다.

| 세그먼트             | 정의                                 | 분석 접근                                                                    |
| ---------------- | ---------------------------------- | ------------------------------------------------------------------------ |
| **FC 자사**        | SFN 외식업 몰 이용 중인 프랜차이즈              | 외식업 몰 SKU 풀 활용, 카니발리제이션 방지                                               |
| **FC 타사 (복수매장)** | SFN에 동일 브랜드 매장 2개+ 존재, 유니크 SKU N개+ | 매장 간 교차 비교로 미구매 SKU 즉시 도출. **RSM을 통한 샘플 리스트 선작업 진행 필요** [SFN Comment 반영] |
| **일반 식당**        | FC 단일매장 + 비프랜차이즈 독립 매장             | 메뉴 기반 식자재 추출 + 유사 고객군 비교                                                 |

> **FC 타사 단일매장과 개인 식당의 통합:** FC 타사이더라도 SFN에 단일 매장만 존재하면 타 매장과의 교차 비교가 불가능하므로, 분석 접근이 개인 식당과 동일하다. 따라서 "일반 식당" 세그먼트로 통합 운영한다.

> **FC 타사 복수매장 기준:** 동일 브랜드 전체 매장의 합집합 유니크 SKU 수가 기준 미달인 경우 "일반 식당"으로 간주한다. 기준값은 실데이터 분포 분석 후 확정한다 (기존 12개 기준은 너무 적으므로 상향 필요).

> **샵인샵/복수브랜드 매장 처리:** [FC 매핑 데이터 반영] 44개 본사가 복수 브랜드를 운영하며, 일부 매장은 2개 이상 브랜드를 동시 운영(예: 족발야시장&무청감자탕 67개, 쥬씨차얌 107개). 이러한 복수 브랜드 매장은 **대분류(본사) 코드 기준으로 FC자사 귀속**하되, 클러스터링 시에는 **주력 브랜드(SFN 구매 매출 비중이 높은 브랜드)의 업종을 기준으로 배정**한다. 메뉴 프로파일은 소속 브랜드 전체의 메뉴를 병합하여 구성한다. ※ "주력 브랜드의 업종 기준 배정"이란: 한 매장이 족발+감자탕 2개 브랜드를 운영할 때, SFN 구매 데이터에서 매출 비중이 높은 쪽의 업종(예: 한식-족발)을 해당 매장의 클러스터 배정 기준으로 사용한다는 의미.

> **SFN 기존 세그먼트와의 매핑 관계:**
>
> SFN은 기존 쿠폰 운영 목적으로 Activate / Volume / Core / Growth / 이탈(위험·확실·장기) / 체리피커 세그먼트를 사용하고 있다. 위의 3개 세그먼트(FC자사/FC타사/일반식당)는 **추천 모델용** 분류이며, SFN 기존 세그먼트는 **쿠폰 운영용** 분류이다. 두 분류 체계는 목적이 다르므로 병행 운영하되, 본 프로젝트의 분위수 회귀 모델은 SFN이 To-be로 지향하는 "구매금액과 구매주기의 개별성을 반영한 개인화 쿠폰 발행"을 구현하는 수단이 된다.

### 2.4 유사 고객군 클러스터링

**적용 대상:** "일반 식당" 세그먼트에만 해당한다. FC 자사는 SFN이 지정한 SKU Pool이 존재하고, FC 타사 복수매장은 동일 브랜드 타 매장이 곧 가장 정확한 유사 고객군이므로 클러스터링이 불필요하다.

**클러스터링 축:**

- **업종** (네이버 플레이스 업종 기준, SFN 회원정보 업종은 보조 폴백) — 주축
- **메뉴 가격대** (저가/중가/고가, 단계 수는 데이터 취합 후 재검토) — 주축
- **FC 여부** — 주축 (FC 단일매장과 개인식당을 동일 세그먼트로 통합했으나, FC 여부에 따라 식자재 특성이 다름: 반제품 vs 원재료)


**Phase 1 — 1차 클러스터: 룰 베이스 결정론적 배정**

```
cluster_key = (industry, menu_price_tier, is_franchise, revenue_size_tier)

예시:
  한식 × 중가 × 비FC × M  → cluster_001
  한식 × 중가 × FC단일 × L → cluster_002
  일식 × 고가 × 비FC × L  → cluster_003

→ 클러스터당 최소 30명 유지, 미달 시 상위 클러스터로 병합
```

**Phase 1 — 2차 서브클러스터: ML 기반 세분화 (TF-IDF + K-Means)**

기존 자카드 유사도의 한계(식자재 중요도·양·맥락 미반영)를 극복하기 위해, Phase 1부터 ML 클러스터링을 도입한다.

```
[1차] 룰 베이스 클러스터 배정: 업종 × 가격대 × FC여부 × 규모
    → 한식 × 중가 × 비FC × M 클러스터 (예: 120명)

[2차] TF-IDF + K-Means 서브클러스터링:
    각 고객의 식자재 벡터 구성 (LLM 추출 + 실제 구매 이력 결합)
    → TF-IDF 벡터화 (고객 유형별 TF 정의 차등 적용)
    → K-Means 클러스터링으로 서브클러스터 생성
    → 실루엣 스코어 ≥ 0.25 이상이면 채택, 미달 시 세분화 생략

    메뉴 유사도 정밀화:
      전체 메뉴 리스트뿐 아니라 네이버의 "대표"/"인기" 메뉴의
      주재료 유사도를 TF-IDF 벡터에 가중 반영하여 세분화 정밀도를 높인다.
      (한식의 경우 찌개도 팔고 구이도 파는 복합 메뉴가 일반적이므로
       대표 메뉴 기준이 없으면 대부분 미분류(혼합형)로 빠질 위험)

    예시 (실제는 예시와 같이 명확히 구분되지 않을 수 있음):
      서브클러스터 A (찌개류 대표 메뉴): 42명
      서브클러스터 B (구이류 대표 메뉴): 35명
      서브클러스터 C (밥류 대표 메뉴): 28명
      미분류 (실루엣 스코어 미달): 클러스터 전체 패턴 적용

[검증] 이중 검증 체계:
    ├── 정량: 실루엣 스코어, SKU 구매 패턴 유사도 (자카드 대비 개선 여부)
    ├── 정성: SFN 도메인 전문가(영업 담당자) 검수 — "이 묶음이 비즈니스적으로 말이 되는가"
    └── **⚠️ 서브클러스터의 정확도 면밀한 추적 필요** [SFN Comment 반영]
         서브클러스터가 얼마나 정확도 있는 역할을 할지가 관건.
         추천 성과와 서브클러스터 배정 정확도를 지속적으로 모니터링하여
         서브클러스터링의 실효성을 검증한다.

[폴백] K-Means 실루엣 스코어가 전 클러스터 0.25 미만이면 세분화 자체를 생략
       → 5월 런칭에 영향 없음 (클러스터 전체를 하나의 서브클러스터로 운영)

[안정성 제약] [v4.2 추가]
  K-Means는 비결정론적이므로 월 1회 재클러스터링 시 서브클러스터 배정이 변동되어
  추천 Pool, 레퍼런스 고객, 연관구매 스코어가 급변할 수 있음.
  → 안정성 확보 방안:
    ├── 재클러스터링 시 이전 배정과의 ARI(Adjusted Rand Index) 비교
    │   ARI < 0.7이면 재배정 보류, 기존 배정 유지
    ├── 고객별 서브클러스터 소속 확률(soft assignment) 기록
    │   소속 확률 < 0.6이면 경계 고객으로 태깅, 복수 서브클러스터 참조 허용
    └── Phase 2에서 HDBSCAN 전환 시 노이즈 허용으로 안정성 자연 개선
```
<!-- [v4.2 변경근거] 논리검증 B2(High): K-Means 비결정론적 특성으로 월간 재클러스터링 시 고객 배정 변동 → 추천 급변 위험. ARI 기반 안정성 제약 및 soft assignment 도입 -->

> **ML 클러스터링 상세 설계**는 별도 문서에 기술한다: TF 정의 선택지, 고차원 희소 벡터 처리, 최적 K값 선정, 자카드 vs K-Means 비교 실험 등.

> **Phase 2 — ML 고도화 (예시):** TF-IDF → 식자재 임베딩(Word2Vec/FastText), K-Means → HDBSCAN(자동 K, 노이즈 허용), 재학습 주기 월 1회 → 주 1회 자동화.

### 2.5 군집 내 패턴 분석 (추천 Pool 보완 및 용량·주기 추정)

**목적:** 클러스터/서브클러스터 내 구매 패턴을 분석하여 ① 추천 SKU Pool의 보완 (메뉴-식자재 매핑 외 추가 SKU 발견), ② 구매 용량 추정, ③ 구매 주기 추정에 활용한다.

**"VIP" 개념의 재정의:** [SFN 피드백 반영: 매출 상위 20%→구매범위 절대 기준]

기존 v3에서 "VIP를 벤치마크로 삼아 VIP가 사는 SKU를 일반 고객에게 추천"하는 방식이었으나, SFN 피드백을 반영하여 다음과 같이 재정의한다.

| 기존 (v3)                      | 변경 (v4)                                                                                                                      |
| ---------------------------- | ---------------------------------------------------------------------------------------------------------------------------- |
| VIP = 매출 상위 20%              | **구매범위 우수 고객 = 수요 식자재의 30~50%를 차별화상회에서 구매하는 고객. 업종별로 매출 규모 및 구매 SKU가 큰 편차를 보이므로, 업종별로 구분하여 실제 데이터 기반으로 선정** [SFN Comment 반영] |
| VIP 구매 SKU = 추천 SKU Pool의 기준 | 추천 SKU Pool = **메뉴 기반 식자재 매핑(1차)** + **서브클러스터 전체 구매 SKU(2차 보완)**                                                             |
| VIP 패턴 = 모든 분석의 기준선          | 구매범위 우수 고객 패턴 = **용량 추정**과 **주기 추정**의 보조 지표                                                                                  |

**구매범위 우수 고객(이하 "레퍼런스 고객") 선정 기준:**

기존 RFM 기반 매출 상위 20%가 아닌, **"해당 식당의 수요 식자재 중 차별화상회에서의 구매 비율"** 기반 절대 기준을 사용한다.

```
레퍼런스 고객 선정:
  ① 해당 고객의 메뉴 기반 수요 식자재 리스트 산출 (§2.1.1)
  ② 수요 식자재 중 차별화상회에서 최근 90일 내 구매한 비율 산출
  ③ 비율 30~50% 이상이면 레퍼런스 고객으로 선정 (정확한 기준값은 데이터 분석 후 확정)
  ④ 추가 조건: 최근 3개월 이내 활발한 구매 (최근성 확보)
  ⑤ **업종별 차등 적용 필수:** 업종별로 매출 규모 및 구매 SKU 수가 큰 편차를 보이므로, [SFN Comment 반영]
     업종별로 구분하여 실제 데이터를 기반으로 기준값을 선정한다.
     (예: 한식은 SKU 다양성이 높아 30%가 적절할 수 있으나, 카페는 50%가 적절할 수 있음)

선정 기준의 근거:
  매출이 높다고 다양한 식자재를 차별화상회에서 구매하는 것은 아님.
  차별화상회에서의 구매 범위가 넓은 고객이야말로
  "이 업종에서 차별화상회가 커버할 수 있는 SKU 유니버스"를 가장 잘 보여주는 고객임.
```

**레퍼런스 고객 패턴 활용 범위:**

| 활용 영역              | 활용 방식                                             | 의미     |
| ------------------ | ------------------------------------------------- | ------ |
| ~~추천 SKU Pool 결정~~ | **사용하지 않음** — SKU Pool은 메뉴 기반 + 서브클러스터 전체 구매 Pool | -      |
| **용량 추정**          | 유사 메뉴·유사 매출 규모의 레퍼런스 고객 구매 용량 참고                  | 의미 있음  |
| **주기 추정**          | 유사 메뉴·유사 매출 규모의 레퍼런스 고객 재구매 간격 참고                 | 의미 있음  |
| **브랜드/원산지/가공수준**   | **개인화 영역** — 레퍼런스 고객이 아닌 해당 고객 자신의 구매 이력 기반       | §3.3.3 |

**서브클러스터 전체 구매 SKU 활용:**

추천 SKU Pool 보완 시 레퍼런스 고객의 구매 SKU가 아닌 **서브클러스터 전체 고객의 구매 SKU**를 사용한다. 레퍼런스 고객의 구매 SKU는 서브클러스터 전체 구매 SKU의 부분집합이며, 소수 샘플(레퍼런스 고객)의 구매 패턴보다 전체 고객의 구매 패턴이 더 넓은 SKU 커버리지를 제공한다.

```
추천 SKU Pool 구성 (일반 식당 세그먼트):
  [1차] 메뉴 기반 식자재 → SFN SKU 매핑 (핵심)
  [2차] 서브클러스터 전체 구매 SKU 중 해당 고객 미구매 SKU (보완)
  [제거] 이미 구매 중인 SKU, 대체품 관계 SKU 제거

서브클러스터 내 SKU별 구매 매장 비중이 높은 SKU를 우선순위화:
  → 해당 서브클러스터에서 구매 매장 비중 = (해당 SKU 구매 매장 수) / (전체 매장 수)
  → 비중이 높을수록 해당 세그먼트에서 수요가 검증된 SKU

⚠️ 침투율 사용 범위 제한 (매튜 효과 방지): [v4.2 추가]
  "서브클러스터 내 SKU별 구매 매장 비중(침투율)"이 §2.5(Pool 보완), §2.6(티어 분류),
  §3.3.2(수요량 스코어 w2)에서 3중으로 사용되면, 침투율 높은 SKU가 과도하게 유리해지고
  침투율 낮은 SKU는 추천 기회 자체가 박탈된다 (이미 잘 팔리는 상품만 더 추천하는 현상).
  → **Pool 보완(§2.5)에서는 침투율을 우선순위화에 사용하되,
     수요량 스코어(§3.3.2)에서는 침투율 항(β × 구매_매장_비중)을 제거**하고,
     대신 레퍼런스 고객 구매 빈도(γ)와 메뉴 투입 빈도(α)만으로 산출한다.
  → §2.6 티어 분류는 Pool 포함/제외 결정 용도이므로 유지하되,
     30% 미만 제외 기준은 Phase 1 운영 데이터 축적 후 적정 기준값 재검증한다.
```

### 2.6 갭 분석 (결핍 SKU 도출)

메뉴 기반 식자재 매핑 + 서브클러스터 전체 구매 패턴을 결합하여, 대상 고객이 아직 구매하지 않는 SKU(결핍 SKU)를 도출한다.

**갭 분석 로직:**

```
[메뉴 기반 수요 식자재 + 서브클러스터 전체 구매 SKU]
    vs
[대상 고객의 실제 구매 이력]
    =
[결핍 SKU 리스트 + 결핍 유형]
```

**침투율 기반 추천 우선순위 티어:** [SFN 피드백 반영]

서브클러스터 내 SKU별 침투율(구매 매장 비중)을 기준으로 결핍 SKU의 추천 우선순위를 4단계로 분류한다.

| 침투율 | 티어 | 의미 | 추천 정책 |
|--------|------|------|-----------|
| **80%+** | 최우선 추천 | 거의 필수 식자재 — 이 고객만 안 사고 있음 | 즉시 추천 + 상품 쿠폰 적극 부착 |
| **50~80%** | 우선 추천 | 과반 이상이 구매하는 주요 식자재 | 추천 풀 상위 배치 |
| **30~50%** | 일반 추천 | 보편적이지는 않으나 유의미한 수요 | 추천 풀 포함, 일반 순위 |
| **30% 미만** | 추천 대상 제외 | 보편적 수요 아님 (특정 매장 특화 SKU) | 추천 풀에서 제외 |

**결핍 유형 분류:**

| 결핍 유형 | 정의 | 예시 | 액션 |
|-----------|------|------|------|
| **완전 미구매** | 수요 식자재인데 한 번도 안 산 SKU | 김치찌개집인데 다시다를 안 삼 | 상품 추천 + 상품 쿠폰 |
| **빈도 부족** | 사긴 사는데 서브클러스터 평균 대비 구매 주기가 길음 | 평균은 주 3회, 이 고객은 월 2회 | 구매 주기 도래 시 리마인드 추천 |
| **용량 부족** | 같은 SKU를 사지만 소량만 구매 | 레퍼런스 고객은 10kg, 이 고객은 2kg | 대용량 SKU 추천 + 단가 메리트 강조 |
| **카테고리 부족** | 특정 카테고리 전체를 안 삼 (완전 미구매의 하위 집합) | 조미료 카테고리 구매 전무 | 카테고리 입문 SKU 추천 |

**FC 타사 복수매장의 갭 분석 (가장 정확):**

동일 브랜드의 A매장이 사는데 B매장이 안 사는 SKU를 직접 비교한다. 같은 메뉴를 파는 같은 브랜드이므로 **수요 식자재가 동일**하다 (단, 매장별 매출이 상이하므로 식자재 필요량 자체는 다를 수 있음).

```
A매장 구매 SKU: {양파, 당근, 돼지고기, 고추장, 참기름, 다시다}
B매장 구매 SKU: {양파, 당근, 돼지고기, 고추장}
                                                    ↑
                         결핍 SKU = {참기름, 다시다} → 추천 대상
```

**용량 비교 기준 (실데이터 반영):**

실데이터에는 `unit_size` 단일 컬럼이 없으므로, 용량 비교 시 `낱개규격 + 낱개규격 UOM + 입수` 조합으로 단위를 통일하여 비교한다.

### 2.7 구매 주기 모델링 [SFN 피드백 반영: 추천 타이밍 이중 구조 명확화]

고객별·SKU별 "언제 살 것인가"를 예측한다. 음식점은 식자재를 반복 구매하는 특성이 있어(실데이터 확인: 고객당 평균 12.45회 주문) 구매 주기 예측의 정확도가 높다.

**구매 주기 모델링의 활용 영역 명확화:**

- **상품 추천 자체는 Daily로 진행된다** — 고객이 자발적으로 사이트에 접속할 때 항상 추천 상품이 노출되어야 하므로, 모든 고객에게 매일 추천 상품 + 상품 할인 쿠폰이 준비되어야 한다.
- **구매 주기 모델링은 LMS(문자) 발송 타이밍 결정에 사용된다** — 개인별 구매 주기 기준으로 "재구매가 필요한 상품" + "개인화 추천 상품 w/상품할인쿠폰" + "개인화 장바구니 쿠폰"을 LMS로 발송한다.

**방법 1 — 통계 기반 구매 인터벌 (Phase 1):**

- 고객별·SKU별 과거 구매 간격(interval)을 계산
- 중앙값 또는 가중 평균을 "예상 구매 주기"로 설정
- 마지막 구매일 + 예상 구매 주기 = 다음 구매 예상일
- 해당 고객의 이력이 부족한 경우 서브클러스터의 SKU별 평균 구매 주기를 적용
- `stock_type`(저장 STORED 61.9% / 일배 NOT_STORED 38.1%)을 구매 주기 모델의 설명변수로 추가: 일배품은 구매 주기가 짧을 가능성이 높음

**방법 2 — 콕스 비례 위험 모델 (Phase 1 후반~Phase 2):**

의학의 생존 분석에서 차용한 방법으로, "이 고객이 이 SKU를 언제 다시 살 것인가"를 확률적으로 예측한다.

- 종속변수: 재구매까지의 시간 (time-to-event)
- 공변량: 고객 세그먼트, 업종, SKU 카테고리, 과거 구매 빈도, 계절성, stock_type, 보관방법
- 산출: 고객×SKU별 "향후 7일 내 구매 확률"

> **Phase 배치:** Phase 1에서는 통계 기반 구매 인터벌을 사용하고, 데이터 축적 후 콕스 비례 위험 모델로 전환한다. 기존 2년치 구매 데이터를 먼저 분석하여 구매 주기 패턴의 규칙성을 검증한 후 적용 여부를 SFN과 공동 판단한다.

### 2.8 장바구니 쿠폰 반응 분석

기존 장바구니 쿠폰 사용 이력 데이터를 활용하여 고객별 쿠폰 반응 패턴을 분석한다.

**실데이터 기반 범위 조정:**

쿠폰 마스터/발급이력 테이블이 미제공된 상태이므로, "발급 대비 사용률" 산출이 불가능하다. Phase 1에서는 **구매 시 쿠폰 동반율 + 바스켓 사이즈 변화** 기반으로 분석 범위를 축소 운영한다.

**가능한 분석 (현재 데이터):**

| 지표 | 산출 방법 | 용도 |
|------|-----------|------|
| 쿠폰 동반율 | 쿠폰 사용 주문 / 전체 주문 (현재 12.8%) | 쿠폰 민감/둔감 초기 분류 |
| 바스켓 사이즈 변화 | 쿠폰 사용 시 결제금액 vs 미사용 시 결제금액 | 쿠폰 효과 측정 |
| 쿠폰 유발 추가 구매액 | (쿠폰 사용 시 결제금액) − (미사용 시 평균 결제금액) | 순 증분 효과 |
| 연속 쿠폰 사용 | 최근 5회 주문 시 쿠폰 사용 빈도 | 체리피커 판정 (제한적) |

**고객 쿠폰 반응 유형 분류 (현재 데이터 기준):**

| 유형 | 판정 기준 | 대응 |
|------|----------|------|
| **쿠폰 활용형** | 쿠폰 동반율 높고 바스켓 사이즈도 증가 | 정상 발급 유지 |
| **쿠폰 의존형 (체리피커)** | 쿠폰 동반율 높지만 바스켓 증가 없음. SFN 기준: 최근 5회 구매 시 100% 쿠폰 사용 | 쿠폰 비용율 하향 발급 |
| **쿠폰 무관심형** | 쿠폰 동반율 낮고 구매 패턴 변화 없음 | 쿠폰 발급 유지, **장바구니 쿠폰 안내 LMS 미발송** (CS 방지), **상품 추천 LMS는 정상 발송** | <!-- [v4.2 변경근거] 회의록 안건 7: 쿠폰 무관심형에 대해 장바구니 쿠폰 LMS만 미발송하고 상품 추천 LMS는 발송하기로 결정 -->
| **쿠폰 반응형** | 쿠폰 동반율은 낮지만 사용 시 바스켓 증가 뚜렷 | 타이밍 최적화로 효과 극대화 |

> **쿠폰 마스터 확보 후 전환:** SFN에 쿠폰 마스터/발급이력 테이블(P0 요청)을 확보하면 "발급 대비 사용률" 기반 정밀 분석으로 전환하고, 장바구니/상품 쿠폰 구분 분석도 가능해진다.


### 2.11 우식이(저장 상품) 기반 분석 [SFN 피드백 반영]

SFN의 "우식이" 기능에서 고객이 직접 담은 상품 + 구매한 상품 데이터를 추천에 활용한다. 우식이 데이터는 **고객이 명시적으로 관심을 표현한 상품**이므로 높은 구매 전환 가능성을 가진다.

**우식이 기반 추천 규칙:**

```
[추천 대상]
  ├─ 우식이에 담았는데 해당 고객의 평균 구매주기 초과 시 → 추천
  └─ 우식이에 담았는데 최근 30일 미구매 상품 → 추천

[추천 제외]
  ├─ 카테고리 평균 구매주기 대비 N배 초과 (시즌 상품, 메뉴 변경 방지)

  고객 A가 우식이에 "참기름 1.8L"을 담아두었으나

```

## 3. 상품 추천 방법론

### 3.1 추천의 목적

상품 추천의 목적은 단순히 "좋은 상품"을 보여주는 것이 아니라, **이 고객이 사야 하는데 아직 안 사고 있는 SKU**를 찾아 구매로 전환시키는 것이다. 이를 통해 고객당 주문당 구매 SKU 수를 5개에서 7~8개로 확대하고, 주문당 바스켓 사이즈를 키우며, 순 GP 마진 13%를 유지한다.

**추천 노출 방식:**

- 상품 추천은 **Daily**로 재계산되며, 고객이 사이트에 접속할 때 항상 추천 상품이 노출된다.
- 추천 상품에 상품 할인 쿠폰이 부착되는 **조건**: [SFN Comment 반영]
  - **계근 상품(standard_type = NON_STANDARD)은 할인에서 제외**
  - **경쟁사 가격 비교에서 SFN이 더 저렴하거나 유사 수준이면 추천만 하고 상품할인쿠폰은 미부착**
  - 경쟁사 대비 가격 열위인 경우에만 상품 할인 쿠폰 부착
- 구매 주기 기반 LMS 발송(주 1회 수준)과 별개로, 사이트 내 추천은 상시 운영된다.

**신규 고객 정책:** [SFN 피드백 반영: 익일→실시간 처리]


### 3.2 SKU 풀 생성 (고객 유형별)

추천 가능한 SKU의 범위를 먼저 특정한다. B2C와 달리 B2B 식자재는 고객당 SKU 풀이 유한하다(500~2,000개). 따라서 복잡한 딥러닝보다 통계 기반 룰이 더 효과적이다.

> **SKU 풀 생성 대원칙:** 모든 고객 유형에서 SKU 풀의 1차 기준은 **"고객의 메뉴-식자재 매핑"**이다. 서브클러스터 전체 구매 패턴은 SKU 풀 **보완**에 활용하며, 풀 내 우선순위는 §3.3 스코어링으로 결정한다.

**FC 자사 고객:**

```
SFN이 프랜차이즈 브랜드별로 선정하여 전달하는 SKU Pool
    (전달 시점: 차별화상회 고객 프로파일링 완료 후 —
     브랜드별/SKU별 GP 기여도 및 구매 매장 비중 분석 필요)
    − 이미 SFN에서 구매 중인 SKU
    − 외식업 몰에서 잘 팔리는 SKU는 SKU Pool 단계에서 선제 제거 (카니발리제이션 방지)
      **"잘 팔리는" 기준 정의:** 외식업 몰 내 해당 브랜드 매장의 월 평균 주문 건수 상위 50% SKU
      (기준값은 SFN 외식업 몰 판매 데이터 분석 후 확정) [v4.2 추가] <!-- [v4.2 변경근거] 논리검증 C1(Low): "잘 팔리는"의 기준이 미정의되어 Pool 제거 vs score×0.3 적용의 경계가 불명확 -->
    = 추천 후보 SKU 풀
```

**FC 타사 복수매장:**

FC 복수매장의 경우, 타 매장 구매 이력 자체가 "그 브랜드 메뉴에 필요한 식자재"이므로 메뉴-식자재 프레임과 일치한다.

```

    − 대상 매장이 이미 구매 중인 SKU
    − 대체품 관계 SKU
    = 추천 후보 SKU 풀
```

### 3.3 우선순위 스코어링

SKU 풀 내에서 어떤 상품을 먼저 추천할지를 결정하는 스코어링 체계다. §3.2에서 SKU Pool이 확정된 후, 모든 논의는 **SKU 수준**에서 진행된다.

**5축 통합 스코어 공식:**

```
final_score(C, t) = w1 × 연관구매 스코어
                  + w2 × 수요량 스코어
                  + w3 × 메뉴프로파일 스코어
                  + w4 × 회사베네핏 스코어
                  + w5 × 가격경쟁력 스코어
```

> **v4 대비 변경:**
> - 기존 `w2(규모프로파일)` → `w2(수요량)`: 매출 규모가 아닌 수요량 기반으로 변경 (어떤 SKU를 살지는 메뉴에 달려있지 매출 규모와 연계되지 않음)
> - **`w5(가격경쟁력)` 신규 추가**: SFN이 가장 강하게 지적한 누락 항목. 경쟁사 대비 당사 판가의 경쟁력이 추천 우선순위에 반드시 반영되어야 함

**고객 유형별 가중치 테이블:**

| 고객 유형                | w1 (연관) | w2 (수요량) | w3 (메뉴) | w4 (회사) | w5 (가격경쟁력) | 근거                            |
| -------------------- | ------- | -------- | ------- | ------- | ---------- | ----------------------------- |
| **기존 고객 (구매 이력 풍부)** | 0.30    | 0.15     | 0.10    | 0.20    | 0.25       | 실제 구매 패턴 + 가격경쟁력              |
| **기존 고객 (구매 이력 빈약)** | 0.10    | 0.20     | 0.25    | 0.20    | 0.25       | 프로파일 기반 보완                    |
| **신규 고객**            | 0.00    | 0.25     | 0.30    | 0.20    | 0.25       | 구매 이력 없음, 메뉴+수요량 핵심           |
| **FC 자사 고객**         | 0.05    | 0.05     | 0.05    | 0.35    | 0.50       | SFN 지정 Pool, **가격경쟁력이 가장 중요** |
| **FC 타사 복수매장**       | 0.35    | 0.10     | 0.10    | 0.20    | 0.25       | 동일 브랜드 타매장 이력 핵심              |

**구매 이력 풍부/빈약 기준:** 최근 60일 내 유니크 SKU 5개 이상 구매 = 풍부, 미만 = 빈약 (기존 90일에서 60일로 단축)

> **FC 자사 참고:** SFN이 지정한 SKU Pool이 있으므로 w1/w2/w3의 비중이 매우 낮다. 외식업 몰에서 안 사는 SKU는 대부분 가격 경쟁력이 없어서이므로, **가격경쟁력(w5)이 가장 중요**하다. **FC자사 고객의 w5 가격경쟁력 비교 대상은 외부 경쟁사(식봄/배민상회)이며, SFN 외식업 몰 가격은 비교 대상이 아니다.** FC자사 고객은 외식업 몰과 차별화상회 양쪽 접근 가능하나, w5는 "외부 경쟁사 대비 SFN 가격의 경쟁력"만을 측정한다. 외식업 몰 ↔ 차별화상회 간 내부 가격 차이는 w5가 아닌 SKU Pool 단계(§3.2)에서 이미 처리된다 (외식업 몰에서 잘 팔리는 SKU는 Pool에서 선제 제거). <!-- [v4.2 변경근거] 논리검증 B3(Medium): FC자사 w5 가격경쟁력의 비교 대상(외식업 몰 vs 차별화상회 vs 경쟁사)이 미정의. 외부 경쟁사 비교로 명시하여 모호성 해소 -->

> **⚠️ 고객 유형 라벨 전달 메커니즘 필요:** [SFN Comment 반영] 고객 유형(기존 고객-이력 풍부/빈약, 신규, FC 자사, FC 타사)은 Weraser에서 분석·분류하는 정보인데, **SFN이 이 고객 구분을 어떻게 인지할 수 있는지** 확인이 필요하다. 고객별로 레이블을 붙여 SFN에 전달하는 경우 SFN에서 어떻게 받아서 처리할 수 있을지 데이터 교환 메커니즘을 확정해야 한다. 실적이 좋지 않으면 고객별 알고리즘(가중치) 수정이 필요하므로 고객 구분 정보의 공유가 필수적이다.

#### 3.3.1 연관구매 스코어 (w1)

"A를 산 고객이 B도 사더라" 패턴을 수치화한다. **서브클러스터 내**에서 상품 간 구매 상관관계를 분석한다 (전체 고객 기반이 아님).

```
산출:
  서브클러스터 내 고객들의 최근 60일 구매 데이터에서
  SKU 쌍의 confidence × lift 합산

대체품 충돌 처리:
  A→B 추천 시, 대상 고객이 B의 대체품을 이미 구매 중이면
  §3.4 필터링 단계에서 제거 (연관구매 스코어 산출 자체에서는 제거하지 않음)

연관 분석 카테고리 수준:
  SKU 수준 일대일 매핑이 아닌 L3 카테고리 수준으로도 연관 분석 수행
  → L3이 실무적으로 더 유의미 (L2 "신선채소"는 너무 넓음)
```

#### 3.3.2 수요량 스코어 (w2)

**"이 식당에서 이 식자재의 수요가 얼마나 높은가"**를 수치화한다. 기존 "규모 프로파일"에서 변경 — 매출 규모가 아닌 메뉴 투입 빈도 기반 수요량을 반영한다.

```
수요량 스코어 구성:

① 메뉴 투입 빈도:
   해당 고객의 메뉴 중 몇 개의 메뉴에 이 식자재가 투입되는가
   → 대부분의 메뉴에 투입되는 식자재는 수요량이 높음
   예: 중국집 → 식용유 (볶음이 많아 수요 높음)
       베이커리 → 버터 (빵 제조에 필수, 수요 높음)
       파스타집 → 파스타면 (모든 메뉴에 투입, 수요 높음)

② 서브클러스터 내 구매 매장 비중:
   해당 서브클러스터에서 이 SKU를 구매하는 매장의 비율
   → 비중이 높을수록 해당 세그먼트에서 수요가 검증된 SKU

③ 레퍼런스 고객 구매 빈도:
   같은 서브클러스터 내 레퍼런스 고객이 이 SKU를 얼마나 자주 사는가
   → 빈도 높을수록 해당 업종에서 필수 식자재

수요량_스코어 = α × 메뉴_투입_빈도 + γ × 레퍼런스_구매_빈도 + δ × **계절지수**
  ※ 기존 β × 구매_매장_비중 항은 §2.5 Pool 보완 및 §2.6 티어 분류에서 이미 사용되므로
    매튜 효과 방지를 위해 수요량 스코어에서는 제거 [v4.2 변경 — 논리검증 A2 반영]

④ 계절지수: [SFN Comment 반영]
   해당 SKU(또는 식자재)의 계절별 수요 변동을 반영한다.
   → 과거 구매 데이터에서 월별 구매량 변동 패턴 추출
   → 현재 시점의 계절 수요 승수(seasonal multiplier)를 수요량 스코어에 반영
   예: 여름 → 냉면 재료 수요 ↑, 겨울 → 국물 재료 수요 ↑
   ※ w3(메뉴프로파일 스코어)의 계절 보정과는 별도. w2는 SKU 수준의 절대 수요량,
      w3는 메뉴 기반 매핑 확신도에 대한 계절 보정
```

#### 3.3.3 메뉴프로파일 스코어 (w3)

"이런 메뉴를 파는 가게는 이걸 사더라" 패턴을 수치화한다. 크롤링된 메뉴에서 식자재를 추출하고 SKU와 매핑한 뒤, 서브클러스터 내 실제 구매 검증 비율을 산출한다.

```
w3 = (LLM 추출 식자재 매핑 여부) × (서브클러스터 내 구매 검증 비율) × (계절 보정)

계절 보정: 계절 한정 메뉴 태그 보유 → 해당 계절에 스코어 상향
          (계절성 메뉴의 판매 증가에 따른 수요량 증가 반영)

계절 보정 구현 방법: [v4.2 추가]
  → 네이버 플레이스에 계절 메뉴 태그가 별도로 없으므로,
    LLM이 메뉴명에서 계절성을 추론 (예: "냉면", "냉콩국수" → 여름, "설렁탕" → 겨울)
  → 계절 경계: 봄(3~5월), 여름(6~8월), 가을(9~11월), 겨울(12~2월)
  → 과거 2년 구매 데이터의 월별 SKU 판매량 변동 패턴으로 계절지수 산출
```

#### 3.3.4 회사베네핏 스코어 (w4)

이 상품을 팔면 SFN에 얼마나 이득인가. 수요량을 반영하여 "마진 높으면서 수요도 높은 상품"이 최적 추천 대상이 되도록 한다.

```
company_benefit_score(t) =
    0.30 × norm(gp_margin(t))           -- GP 마진 높을수록 유리
  + 0.25 × norm(unit_price(t))          -- 단가 높을수록 매출 기여
  + 0.20 × demand_weight(t)             -- 수요량 반영 (신규)
  + 0.15 × volume_upgrade_flag(t)       -- 대용량 SKU 가산
  + 0.10 × strategic_item_flag(t)       -- 전략 상품 가산 (SFN 지정)

각 항목을 0~1 범위로 정규화(min-max) 후 가중합
```

> **w4(회사베네핏)와 w5(가격경쟁력)의 conflict 해소:** [SFN Comment 반영] GP 마진이 높은 상품이 반드시 가격 경쟁력이 높은 것은 아니다. 두 축의 충돌은 **최종 스코어의 weight(w4, w5 가중치) 배분으로 보정**된다. 예를 들어 FC 자사 고객은 w5=0.50, w4=0.35로 가격경쟁력을 우선하고, 기존 고객(이력 풍부)은 w5=0.25, w4=0.20으로 균형을 유지한다. 특정 상품이 GP는 높으나 가격경쟁력이 낮으면 w5에서 감점되어 최종 스코어가 조정된다.
>
> **고도화 검토 (2026.03.05 회의 — 대표 논의 후 확정 예정):** [v4.2 추가]
> - 고객 세그먼트(로열 고객 vs 이탈 위험 고객)와 상품 특성(전략 상품 vs 일반 상품)에 따라
>   w4/w5 가중치를 차등 적용하는 방안이 제시되었으나, 양사 대표 간 재논의 후 확정 예정
> - §3.3.7 GP 기반 동적 웨이트 조정 로직이 현재 일괄 적용 방식으로 설계되어,
>   고객 유형별·상품 특성별 차별화가 반영되지 않는 점에 대해 구조 재검토 필요

**GP 마진 산출 (실데이터 반영):**

```
GP 마진 = (판매가 - 실구매가) / 판매가

실데이터 현황:
  `실구매가` 기반 GP 산출 가능: 전체 9,265 SKU 중 67.2% (평균 21.7%, 중앙값 23.7%)
  마이너스 마진 상품: 69개 (1.1%)
  문서 목표 GP 15%는 충분히 달성 가능

나머지 32.8% 실구매가 미보유의 실체 [SFN 회신 반영]:
  ├── 대부분: 판매중지/대기 상품 → §3.4에서 이미 추천 제외 (상품상태 ∉ '판매가능')
  ├── 소수: 수급이슈로 ERP 코드 일시중단/변경 → 추천 제외 처리
  │         (1일~수일 내 원복, Daily 상품정보 I/F로 자동 해소)
  └── 결론: '판매가능' 상품 중 실구매가 미보유는 극소수이며,
            해당 상품은 추천 제외 → **판매가능 상품 기준 GP 커버율은 거의 100%**

GP 미커버 폴백 (잔여 극소수 대응):
  ├── 1차: `discount_rate` (정상가→판매가 할인율, 71.8% 커버, 평균 18.1%) 프록시
  ├── 2차: SFN GP 계산 API 호출 (API 연동 후)
  └── 3차: 동일 L3 카테고리 평균 GP 적용
```

**캠페인 기반 전략 상품 태깅:**

#### 3.3.5 가격경쟁력 스코어 (w5) — 신규 [SFN 피드백 반영]

경쟁사 대비 SFN 판매가의 경쟁력을 수치화한다. **가장 중요한 스코어링 축 중 하나**로, SFN 피드백을 반영하여 신규 추가한다.

```
price_competitiveness_score(t) =
    경쟁사 단위용량당 단가 대비 SFN 단위용량당 단가의 우위 정도

산출:
  sfn_unit_price = SFN 판매가 / 단위용량
  competitor_unit_price = min(식봄 단위용량당 단가, 배민상회 단위용량당 단가)

  price_gap = (competitor_unit_price - sfn_unit_price) / competitor_unit_price

  → price_gap > 0: SFN이 더 저렴 → 높은 스코어 (적극 추천)
  → price_gap = 0: 동등 → 중간 스코어
  → price_gap < 0: SFN이 더 비쌈 → 낮은 스코어 (쿠폰 필요)

경쟁사 데이터 없는 SKU:
  → price_competitiveness_score = 0.5 (중립값)
  → 경쟁사 크롤링 범위 확대에 따라 커버리지 향상
```

#### 3.3.6 개인화 보정 (필터링 단계 반영) [SFN 피드백 반영: VIP 패턴→개인 이력 기반 변경]

§3.3.1~3.3.5까지 완료된 상태에서, 단일 식자재에 spec(용량, 원산지, 브랜드, 가공수준 등)이 다른 복수의 SKU가 매핑된 상태이다. 개인화 보정은 이 복수 SKU 중 해당 고객에게 가장 적합한 SKU를 선택하는 단계이다. ※ 원산지·제조사 정보는 **상품마스터 조인 기준**으로 사용한다 (주문상품 데이터 Result 1에는 해당 필드 미보유, 상품마스터 `product_id` 조인율 100%로 완전 보완 가능 확인 완료) [SFN 회신 반영]

| 요소 | 산출 방법 | 반영 방식 |
|------|-----------|-----------|
| 원산지 선호도 | **해당 고객 자신의** 과거 구매에서 국산/수입 비율 | 선호 원산지 SKU에 스코어 보정 |
| 가공수준 선호도 | **해당 고객 자신의** 원물/반가공/완전가공 구매 비율 (보관방법+상품명 파싱) | 선호 가공수준 SKU에 스코어 보정 |
| 용량 비율 추정 | 과거 구매 용량 패턴 + 레퍼런스 고객 용량 참고 | 근접 용량 SKU에 스코어 보정 |
| 브랜드 선호도 | **해당 고객 자신의** 동일 카테고리 내 제조사 점유율. 이력 부족 시 서브클러스터 내 최다 판매 제조사 참고. (예: 오뚜기 케첩 구매 고객 → 홀토마토도 오뚜기 제조사 우선) | 선호 제조사 SKU에 스코어 보정 |

> **핵심:** 브랜드/원산지/가공수준 선호도는 레퍼런스 고객이나 서브클러스터 대표성이 아닌, **해당 고객 자신의 구매 이력**에서 산출한다. 이것은 개인화의 영역이다. 구매 이력 부족 시에만 서브클러스터 내 최다 판매 기준으로 폴백한다.

#### 3.3.7 GP 마진 기반 동적 웨이트 조정

누적 GP 마진 상태에 따라 w4(회사베네핏) **내부의** 마진 vs 매출 가중치를 동적으로 조정한다.

> **동적 웨이트 조정의 핵심 메커니즘 설명:** [SFN Comment 반영: 구두 설명 요청 사항]
>
> w4(회사베네핏) 공식 내부에 `gp_margin`과 `unit_price` 두 항이 있다. 이 둘은 상충 관계다:
> - **gp_margin 가중치 ↑** = 마진율이 높은 상품 우선 (보통 저단가 상품)
> - **unit_price 가중치 ↑** = 단가가 높은 상품 우선 (매출 기여 높음, 마진율은 낮을 수 있음)
>
> 동적 조정이란: **주간/월간 누적 GP 마진 실적을 모니터링하여**, 마진이 목표(13%)보다 높으면 매출 드라이브(고단가 우선)로, 마진이 위험 수준이면 마진 방어(고마진 우선)로 자동 전환하는 것이다.
>
> 예시: 이번 주 누적 GP가 16%라면 → 마진 여유 → unit_price 가중치를 0.40으로 올려 고단가 상품 추천 → 매출 확대. 이번 주 누적 GP가 10%라면 → 마진 위험 → gp_margin 가중치를 0.65로 올려 고마진 상품 추천 → 마진 복구.

| 누적 GP 마진 | 상태 | gp_margin 가중치 | unit_price 가중치 | 전략 |
|--------------|------|-----------------|------------------|------|
| 15% 이상 | 여유 | 0.15 | 0.40 | 매출 드라이브 — 고단가 상품 우선 |
| 13~15% | 정상 | 0.30 | 0.25 | 기본값 — 마진과 매출 균형 |
| 11~13% | 주의 | 0.50 | 0.15 | 마진 방어 — 고마진 상품 우선 |
| 11% 미만 | 위험 | 0.65 | 0.05 | 마진 긴급 복구 |

> 동적 조정 대상은 w4 내부의 gp_margin과 unit_price 가중치만이며 (합계 0.55 고정), demand_weight(0.20), volume_upgrade_flag(0.15), strategic_item_flag(0.10)은 GP 상태와 무관하게 고정이다.

| 추천 2회 노출 후 미구매 | 전체 | score × 0.4 | 관심 낮음, 3회차 미구매 시 제외 필터로 이동 |
| 마이너스 마진 상품 | 전체 | score × 0.1 | 완전 제거하지 않는 이유: 전략적 미끼 상품 활용 가능 |
| ~~최근 반품 이력 카테고리 (품질/포장 외)~~ | ~~전체~~ | ~~score × 0.5~~ | ~~품질/포장 불량 외 반품은 고려하지 않아도 될 것으로 판단~~ [SFN Comment 반영: 품질/포장 외 반품은 물류 귀책이므로 추천에 반영 불필요] |
| **가격경쟁력 매우 낮은 SKU** | 전체 | score × 0.5 | SFN 판가가 경쟁사 대비 20%+ 비싼 경우 [SFN 피드백 반영] |

**다양성 보장 (Diversity Constraint):**

실데이터 카테고리 구조를 반영하여 재설계한다. L1이 5개뿐이고 가공식품이 73.9%를 차지하므로 L1 기준 다양성 제약은 부적절하다.

```
최종 추천 N개 중:
  ├── 동일 category_L2 최대 30% (**신선 카테고리 제외**)
  ├── 동일 category_L3 최대 15% (**신선 카테고리 제외**)
  └── 최소 3개 이상의 서로 다른 category_L2 포함

⚠️ **신선 카테고리는 다양성 제약에서 제외:** [SFN Comment 반영]
  카테고리 다양성은 카테고리 내 중복 구매율이 낮은 **가공식품**의 경우 유효하나,
  **신선(농산물, 수산물, 축산물 등)은 카테고리 내에서의 복수 구매가 정상적**이므로
  다양성 제약을 적용하지 않는다.
  (예: 신선채소 카테고리에서 양파+당근+대파+양배추를 동시 추천하는 것은 정상)

위반 시: 초과 카테고리의 가장 낮은 스코어 SKU를 차순위 카테고리 SKU로 교체 (가공식품에만 적용)

※ 다양성 제약의 목적: 고객이 다양한 카테고리의 상품을 발견하여
   구매 SKU 다양성을 높이는 것. 특정 카테고리만 집중 추천하면
   해당 카테고리 구매는 늘지만 전체 바스켓 확장 효과가 제한됨.
```

### 3.5 로테이션 및 피드백 루프

추천 상품은 1회성이 아니라 지속적으로 교체되며 학습한다.

**로테이션 사이클:**

```
SKU 풀 (FC ~80개, 일반 식당 그 이상)
    → 1회 추천 N개 노출
    → 고객 반응 관찰:
        ├─ 구매 → 추천 성공
        │   → 추천에서는 제외하되, 구매주기 추천은 계속 진행
        │   → 해당 SKU는 정기 구매 모니터링으로 전환
        ├─ 상세페이지 방문 후 미구매 → 관심은 있으나 가격 저항 가능성
        │   → 상품 쿠폰 적용 후 재추천
        ├─ 장바구니 담기 후 미구매 → 가격 저항 → 상품 쿠폰 대상
        └─ 무반응 → 카운터 +1
            ├─ 카운터 < 3 → 다음 로테이션에서 재추천 (가격경쟁력 스코어 재점검)
            └─ 카운터 ≥ 3 → 제외 필터로 이동
                (N = 3, Phase 1 기본값, 백오피스에서 조정 가능)
```

> **행동 데이터 수집 필요:** 상세페이지 방문, 장바구니 담기 등 행동 데이터는 SFN 내부 확인 후 수집 예정 (w/윤정환 실장, 인세용 실장, 정낙정 팀장). 수집 전까지는 "구매 여부"만으로 피드백 루프 운영.

**추천 갱신 주기:**
시스템은 가격 변동 및 재고 상황을 반영하여 **매일(Daily)** 추천을 재계산한다.

**우식이 기반 추천 통합:** [SFN 피드백 반영]
§2.11의 우식이 기반 추천은 일반 추천과 병행하되, 우식이 출처 SKU에 별도 태그를 부여하여 추천 사유를 구분한다. 우식이에 담긴 상품 중 구매주기 초과 or 30일 미구매인 SKU는 로테이션에서 **최우선 노출** 대상으로 설정한다.

## 4. 쿠폰 발행 방법론

### 4.1 쿠폰의 두 가지 유형과 목적

| 구분 | 상품 쿠폰 | 장바구니 쿠폰 |
|------|-----------|--------------|
| 적용 대상 | 개별 SKU | 장바구니 전체 결제 금액 |
| 목적 | 미구매 SKU의 첫 구매 유도 | 바스켓 사이즈 확대 |
| 발급 시점 | 사전 발급 (추천 상품에 부착, 가격 열위 SKU에만) | **선발급** (장바구니 담기 전 사전 지급) [SFN Comment 반영] |
| 핵심 질문 | "이 고객이 이 상품을 살 가능성이 있는가?" | "이 고객에게 적정 할인 금액은 얼마인가?" |
| **할인액 범위** | GP 기반 산정 | **Min 3,000원, Max 제한 없음** [SFN Comment 반영] |
| GP 마진 관리 | 상품별 GP 15% 유지 | 매출 대비 2% 이내 운영 |
| **쿠폰 부착 조건** | 계근 상품 제외, 가격 경쟁력 우위 시 미부착 | SFN이 사전 생성한 쿠폰 Pool에서 배정 |

두 쿠폰을 합산했을 때 전체 GP 마진 13%를 유지하는 것이 제약 조건이다. 또한, SFN이 정의하여 전달하는 **쿠폰 할인율/금액의 최대-최소 범위(Range)** 내에서만 쿠폰을 산정한다.

### 4.2 상품 쿠폰: "살 가능성이 있는 고객에게, 안 사고 있는 상품의 쿠폰을 발행"

#### 4.2.1 대상 고객 × 대상 SKU 선정

상품 쿠폰은 아무에게나 발행하지 않는다. **갭 분석에서 결핍 SKU가 도출되고, 가격 저항이 예상되는 경우에만** 발행한다.

**대상 선정 로직:**

```
Step 1: 갭 분석에서 결핍 SKU가 도출된 고객을 대상으로 선정
    → "메뉴 기반 수요 식자재인데 이 고객은 안 사는 SKU"
    → "서브클러스터에서 구매 매장 비중 높은데 이 고객은 안 사는 SKU"

Step 2: 결핍 유형에 따라 쿠폰 필요성 판단
    → 완전 미구매: 첫 구매 허들이 높음 → 상품 쿠폰 발행 적합
    → 빈도 부족: 이미 구매 경험 있음 → 타이밍 추천으로 충분, 쿠폰 불필요
    → 용량 부족: 대용량 전환 유도 → 대용량 SKU에 쿠폰 발행

Step 3: 가격 경쟁력 기반 쿠폰 부착 판단 [SFN Comment 반영]
    → **계근 상품(NON_STANDARD): 상품 쿠폰 발행 제외** (가격 변동이 크므로)
    → **경쟁사 대비 가격 동등/우위 SKU: 추천만 하고 상품할인쿠폰은 미부착**
    → 경쟁사 대비 가격 열위 SKU (SFN 가격 > 경쟁사 가격): 상품 쿠폰 발행 (가격 갭 해소)
    → **경쟁사 가격 정보 없는 SKU: 가격할인 불필요** (고객도 비교 불가 — §2.1.2 폴백 정책)
    → 상세페이지 방문/장바구니 담기 후 미구매 이력 SKU: 상품 쿠폰 발행 (행동 로그 수집 후)

Step 4: 추가 제외 조건 [v4.2 추가]
    → **마이너스 마진 상품(GP < 0): 추천 대상에는 포함하되, 상품 할인 쿠폰은 발행하지 않음**
      (§3.4 후순위화에서 score × 0.1 적용은 유지, 전략적 미끼 상품 활용 시 예외 허용)
    → **극소액 할인 기준: 경쟁사 가격 비교 결과 할인액이 극소액인 경우 쿠폰 미부착, 상품 노출만 진행**
      (기준값: SFN 정의 예정 — 요율 기준 + 절대 금액 기준)
    → **추가 제약 조건(특정 요일 가격 민감 상품 사용 제한 등): SFN에서 별도 정의하여 전달 예정**
```

#### 4.2.2 할인율 산정 방법

상품 쿠폰의 할인율은 세 가지 제약 조건 안에서 산정한다.

**제약 조건:**

1. **개별 SKU 마이너스 마진 허용:** 쿠폰 적용 후 개별 SKU의 GP 마진이 마이너스가 될 수 있음 (구체적 허용 범위는 SFN 운영 정책에 따름)
2. **전체 GP 15% 유지 (하드 제약):** 상품 쿠폰 적용 후 전체 상품 GP 마진이 15% 이상
3. **경쟁사 가격 수렴:** 할인 후 단위용량당 단가가 경쟁사와 동등하거나 약간 낮은 수준

**GP 데이터별 할인율 산정 (실데이터 반영):**

```
[Case 1] GP 마진 데이터 있는 SKU (판매가능 상품 기준 거의 전수):
  쿠폰 할인액 = min(가격 갭, 판매가 × GP 마진 − 판매가 × 최소 마진)
  → 기존 로직 그대로 적용
  ※ 전체 SKU 중 67.2%이나, 나머지 32.8%는 대부분 판매중지 상품.
    판매가능 상품 중 실구매가 미보유는 극소수이며 추천 제외 처리 [SFN 회신 반영]

[Case 2] GP 마진 없으나 discount_rate 있는 SKU (잔여 극소수):
  discount_rate(정상가→판매가 할인율)를 GP 프록시로 활용
  → 보수적으로 discount_rate × 0.5를 최대 쿠폰 할인율로 제한
  → SFN GP 계산 API 연동 후 실제 GP 기반으로 전환

[Case 3] GP도 discount_rate도 없는 SKU:
  → 동일 L3 카테고리 평균 GP 적용
  → 평균 GP도 없으면 쿠폰 미발행 (추천만 진행)
```

### 4.3 장바구니 쿠폰: "조금만 더 담으면 BEP를 넘길 수 있는 고객에게 발행"

#### 4.3.1 핵심 개념: 분위수 회귀 기반 개인화

> **As-is vs To-be:** 현재 SFN은 LTV 기반 고정 테이블로 운영(As-is). 본 프로젝트의 분위수 회귀 모델은 "구매금액과 구매주기의 개별성을 반영한 개인화"를 구현(To-be). 전환 과도기에는 SFN 고정 테이블을 초기값/폴백으로 사용하고, 분위수 회귀 모델 검증 후 점진적 대체.

**분위수 회귀(Quantile Regression) 모델:**

실데이터 확인: `total_pay_amount` 분포 데이터 충분 (고객당 평균 12.45회 주문).

```
고객 D의 과거 구매 금액 분포:
    25% 분위수 (Q1): 150,000원  ← 4번 중 3번은 이 이상 씀
    50% 분위수 (Q2): 200,000원  ← 절반 확률로 이 이상 씀
    75% 분위수 (Q3): 280,000원  ← 4번 중 1번 정도 이 이상 씀

⚠️ 유의사항: 쿠폰 사용 주문(12.8%)을 포함한 전체 주문 금액 분포로 분위수 산출.
         **쿠폰이 결제금액을 인플레이션시키므로, 쿠폰 미사용 시 추정 금액으로 보정 필요.** [SFN Comment 반영: 분위수 회귀에 쿠폰 사용 주문 포함 시 기준선 왜곡]
         쿠폰 마스터 확보 후 쿠폰 미사용 시 추정 금액으로 보정.

Phase 1 즉시 보정 방안: [v4.2 추가]
  쿠폰 마스터 전면 확보 전이라도, 현재 보유 데이터로 보정 가능:
  ├── 방안 1: 쿠폰 사용 주문(12.8%) 제외 후 분위수 산출 (보수적)
  ├── 방안 2: 쿠폰 사용 주문의 결제금액에서 할인액 차감 후 분위수 산출
  │   (쿠폰 데이터 25.11~26.2 기준: 할인금액 5,000~40,000원, 평균 약 15,000원)
  └── **권장:** 방안 2 우선 적용 — 쿠폰 사용 주문 완전 제거 시 샘플 축소 이슈
```
<!-- [v4.2 변경근거] 논리검증 B1(High): 쿠폰 사용 주문(12.8%)이 바스켓 사이즈를 인위적으로 부풀려 분위수 상향 왜곡. 쿠폰 데이터(5_쿠폰정보 4개 파일) 기준 사용 시 평균 매출 350,948원, 할인금액 범위 5,000~40,000원. Phase 1에서 할인액 차감 보정 즉시 적용 가능 -->

#### 4.3.2 BEP 개념과 쿠폰 금액 산정

**BEP(Break-Even Point):** 쿠폰 할인 비용을 고객의 추가 구매 마진으로 회수하는 손익분기점이다.

```
[기본 시나리오]
  고객 D의 평상시 평균 결제: 200,000원 (50% 분위수)
  GP 마진율: 13%
  평상시 SFN 마진: 200,000 × 0.13 = 26,000원

[쿠폰 발행 시나리오]
  목표 결제 금액: 250,000원 (75% 분위수 근처)
  추가 구매 유도액: 50,000원
  추가 구매 마진: 50,000 × 0.15 = 7,500원 (상품 GP 15% 기준)

  쿠폰 할인액 < 7,500원이면 → 쿠폰 발행이 수익적

  ⚠️ **장바구니 쿠폰은 선발급이므로 실시간 제안 불가:** [SFN Comment 반영]
  "5만원만 더 담으면 3% 할인"과 같은 실시간 제안은 구조적으로 불가.
  따라서 **개별 고객의 쿠폰 민감도 및 반응도를 기반으로 적정 할인 금액을 선계산**해야 한다.
  최초 런칭 시 Rule Setting 필요.

  선발급 쿠폰 설계:
  ├── 고객별 과거 구매 금액 분포 + 쿠폰 반응도 분석
  ├── 적정 할인 금액 사전 산출 (Min 3,000원 ~ Max 제한 없음)
  └── SFN에서 사전에 장바구니 쿠폰을 생성해야 하므로 **전체 쿠폰 Pool 리스트 작성 필요**
```
